# 필상 시스템 — Day 39 고객 지원 운영(SLA/OLA) · 상태페이지 · 커뮤니케이션

1. **날짜**

* 2024-10-02

2. **목표**

* **고객 티켓·문의→진단→해결** 전 과정을 **SLA/OLA**로 표준화
* **상태페이지·알림·RCA 문안** 템플릿 확정, **ko/ja/en** 다국어 운영 정착

---

3. **내용**

### A. 범위 & 원칙

* 범위: 고객 문의(기능/결함/요금) + 인시던트 커뮤니케이션 + 유지보수 공지.
* 원칙: **사실 중심·시간 확정·증거 링크**(대시/EPKG/릴리스 노트) 포함, **ETA 대신 진행상태**로 안내.

### B. 서비스 수준(대외 SLA) & 내부 OLA

| 심각도(고객관점)  | 예시                  | **초기 응답** |   **우선 해결 목표** | 커뮤니케이션 주기 |
| ---------- | ------------------- | --------: | -------------: | --------- |
| **S1(중단)** | 콘솔/API 대규모 장애       |   **15분** | **4h** (복구/우회) | 30분       |
| **S2(중대)** | 인입→사건 지연↑, 일부 기능 저하 |   **30분** |         **8h** | 60분       |
| **S3(보통)** | 특정 테넌트 이슈/설정/오탐 문의  |    **4h** |         **3d** | 1일        |
| **S4(질의)** | 기능요청/가이드            |    **1d** |         **5d** | 필요시       |

* **영업시간**: KR 09:00–18:00 KST / JP 09:00–18:00 JST, **S1/S2 24×7**.
* **OLA(내부)**: SRE/CSIRT 최초 소유자 배정 ≤ **10분**(S1), DBV360 보조판정 ≤ **30분**(S2).

### C. 티켓 라이프사이클(표준)

1. **수신**(포털/메일/웹훅) → 심각도·범주 지정(자동 규칙 + 수동 확인)

.  

2. **소유자 배정**(큐 기반 라운드로빈, S1/S2는 온콜 고정)
3. **진단 & 증거 확보**: 저장 쿼리/EPKG 링크 첨부, 영향/범위 기술
4. **완화/우회** 제시 → **영구 해결** 계획/일정 제안
5. **검증·종결**: 고객 확인 후 Close, **RCA/노트** 링크

* 티켓 필수 필드: `severity, tenant, region, category(billing|incident|howto|fp/miss), purpose, evidence_uri[]`.

### D. 상태페이지 & 공지

* **구성**: 리전별 컴포넌트(KR/JP), 서비스별(콘솔/API/인입/검색/알림) 상태.
* **게시 기준**: S1 즉시 / S2 10분 지속 시 / 계획 점검은 3영업일 전 사전 공지.
* **언어**: ko/ja/en 동시 게시, 공지에는 **타임라인(UTC+지역시각 병기)**, 영향(에러율/지연), 다음 업데이트 시각 명시.

**템플릿(요약)**

* 제목: `[KR] Incident: Ingest→Incident latency ↑`
* 본문: *“2025-10-02 10:14 KST부터 KR 인입→사건 p95가 18s로 상승했습니다. 원인 조사 중이며, 샘플링 완화와 소비자 증설을 적용했습니다. 다음 업데이트는 10:45 KST에 제공하겠습니다. (증거: 대시보드 링크, 변경 타임라인)”*

### E. RCA(대외 요약) 템플릿

* **개요**: 기간/영향 범위(지표·%)/영향 고객
* **원인**: 직접 원인 + 기여 요인(룰/샤드/배포/의존성)
* **완화/복구**: 즉시 조치·영구 조치(마감일 포함)
* **예방**: 테스트/가드레일/플래그/용량 수정
* **증거**: 대시/EPKG(서명 URL 10분), 커밋/릴리스 노트
* **연락처**: 지원 채널/CSM

### F. 커뮤니케이션 매크로(ko/ja/en)

* **S1 초기 안내(ko)**: “현재 KR 리전에서 서비스 지연을 인지했습니다. 즉시 완화 조치를 적용했으며, 30분 주기로 진행 상황을 공유드리겠습니다. 티켓 ID: {id}, 영향: {지표 링크}.”
* **S3 FP 문의 접수(ja)**: “誤検知の可能性を確認しました。比較サンプルとTIスコアを収集し、DBV360にて再判定します（所要30分目安）。結果は本チケットでご連絡します。”
* **해결 안내(en)**: “Issue mitigated and verified via synthetic checks. Root cause and preventive actions are documented here: {RCA link}.”

### G. 도구·연동

* **Ticketing** ↔ **Incident** 양방향 싱크(상태/심각도/증거 링크).
* **Notifier** 정책: 고객 알림은 **테넌트 언어/플랜**별 채널 라우팅.
* **상태페이지 API**: `POST /v1/status/incidents`, `PATCH /v1/status/{id}`(다국어 필드 포함).
* **CS 지표**: `first_response_time`, `resolution_time`, `update_interval_compliance`, `csat`.

### H. 품질/준수 지표 & 가드레일

* **SLA 충족률**: S1 98%/S2 96%/S3 90% 이상.
* **업데이트 준수**: 상태페이지 갱신 간격 준수율 ≥ **95%**.
* **RCA 리드타임**: S1 48h/S2 72h 내 게시.
* **CSAT 목표**: ≥ **4.5/5.0**.
* **보안/주권**: 고객 커뮤니케이션에 **원본 로그 미포함**, 서명 URL·마스킹만 허용.

### I. 운영 체크리스트

* [ ] 티켓 템플릿 자동 채움(테넌트/리전/증거 링크)
* [ ] 상태페이지 다국어 초안 자동 생성(검수 후 게시)
* [ ] RCA 서명·해시체인 고정, 릴리스/가드레일 이슈와 연결
* [ ] 월간 SLA 리포트 발행(테넌트별/리전별)
* [ ] 매크로/용어집 분기 재검토(ko/ja/en 톤&매너 일치)

4. **예시 설명 (S2—KR 지연 이슈: 72분 내 해결·RCA 36h 게시)**

1) 10:14 KST 알람 → **S2로 티켓 생성**, 상태페이지 10:20 게시(ko/ja/en).
2) 증거 패키지 자동 첨부(대시·릴리스 타임라인·DBV360 샘플).
3) 10:27 완화(소비자 증설·샘플링 완화) → 30분 주기 업데이트 준수.
4) 11:26 지연 정상화 확인 → 고객 알림·티켓 **Resolved**.
5) 36시간 내 RCA 게시(원인: 샤드 편향+룰 리플레이 비용 증가, 예방: 샤드 정책/쿼리 캐시 TTL 조정/게이트 추가), CSAT 4.8/5.0.

> 결과: **SLA/OLA·상태페이지·템플릿**이 결합되어 커뮤니케이션이 **신속·일관·검증가능**해지고, 증거 중심 운영으로 신뢰와 만족도를 동시에 높입니다.
