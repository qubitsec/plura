# MSA 개발을 위한 사전 설명 문서

## 1일차 – 개념 이해

**날짜: 6월 10일**

---

## 1️⃣ 모놀리식(Monolithic)이란 무엇인가

**모놀리식 아키텍처**(Monolithic Architecture)란
하나의 애플리케이션이 **단일 코드베이스**, **단일 배포 단위**, **단일 실행 프로세스**로 구성된 구조를 의미합니다.

즉,
**기능·로직·데이터 접근·UI가 하나의 덩어리로 결합된 구조**입니다.

---

## 2️⃣ 모놀리식 구조의 기본 특징

모놀리식 애플리케이션은 다음과 같은 특징을 가집니다.

### 단일 애플리케이션 구조

* 사용자 인증
* 비즈니스 로직
* 데이터 처리
* UI 렌더링

---

### 🔹 단일 빌드 · 단일 배포

* 코드 일부만 수정해도
  ➜ **전체 애플리케이션을 다시 빌드·배포**
* 장애 발생 시
  ➜ **전체 서비스에 영향**

---

### 🔹 강한 결합도 (Tight Coupling)

* 각 기능이 서로 강하게 연결
* 한 모듈의 변경이 다른 모듈에 연쇄 영향

---

## 3️⃣ 모놀리식 아키텍처 예시 구조

일반적인 모놀리식 구조는 다음과 같습니다.

```
[ 사용자 요청 ]
        ↓
[ Web / API Controller ]
        ↓
[ Business Logic ]
        ↓
[ Data Access Layer ]
        ↓
[ Database ]
```

* 모든 계층이 **하나의 프로세스**에서 동작
* 내부 호출은 **메모리 기반 함수 호출**

---

## 4️⃣ 모놀리식의 장점

초기에는 매우 현실적인 선택입니다.

### ✅ 단순한 구조

* 이해하기 쉽고 설계가 직관적
* 개발 진입 장벽이 낮음

### ✅ 빠른 초기 개발

* PoC, MVP 단계에 유리
* 의사결정과 구현 속도가 빠름

### ✅ 운영 단순성

* 서버 수 적음
* 배포·모니터링 구조가 단순

---

## 5️⃣ 모놀리식의 한계

시스템이 성장할수록 한계가 명확해집니다.

### 확장성의 한계

* 특정 기능만 트래픽 증가해도
  ➜ **전체 애플리케이션을 확장해야 함**

### 변경 리스크 증가

* 작은 수정이 전체 장애로 이어질 수 있음
* 배포 시 다운타임 발생 가능성 증가

### 팀 확장의 어려움

* 여러 팀이 하나의 코드베이스를 공유
* 충돌, 의존성, 릴리즈 조율 문제 발생

### 장애 전파

* 하나의 오류가
  ➜ **전체 서비스 장애로 확산**

---

## 6️⃣ 왜 모놀리식에서 MSA를 이야기하게 되는가

대부분의 시스템은 다음 경로를 따릅니다.

> **모놀리식 → 복잡도 증가 → 운영 부담 증가 → MSA 검토**

즉,

* 모놀리식은 **잘못된 구조가 아니라**
* **성장 단계에 맞는 구조**

문제는
**성장 이후에도 그대로 유지할 때 발생**합니다.

---

## 7️⃣ 정리

| 구분     | 모놀리식      |
| ------ | --------- |
| 구조     | 단일 애플리케이션 |
| 배포     | 전체 배포     |
| 결합도    | 매우 높음     |
| 초기 개발  | 매우 빠름     |
| 확장성    | 제한적       |
| 대규모 운영 | 어려움       |
