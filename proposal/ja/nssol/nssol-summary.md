# 1. なぜ今、この提案が必要なのか

日本の製造業とサプライチェーンのセキュリティは、  
個別の製品や機能を追加していくだけでは、実効性ある対応体制を構築しにくい段階に入っています。

課題は、単にセキュリティ機能が不足していることではありません。  
実際の現場では、むしろ次のような問題が大きくなっています。

- Web、アカウント、サーバ、情報持ち出し、対応体制が分散している
- 攻撃は一連の流れで進行する一方、運用は複数のコンソールに分かれている
- 検知後の対応やフォレンジックまで一気通貫でつなげにくい
- サプライチェーン全体で見ると、本社と協力会社の運用レベルにばらつきがある
- とくに地方の中堅・中小製造業では、人材確保と専門性の維持が難しい

つまり、

> **セキュリティの必要性は高い一方で、実際に運用できる対応体制が不足している**  
> という構造的な課題があります。

現在の攻撃は、単発イベントで終わりません。

- Webアプリケーション攻撃
- クレデンシャルスタッフィング
- 認証回避・アカウント侵害
- サーバ上での不審挙動
- 情報持ち出しの試み
- ランサムウェア
- 長期潜伏型APT

したがって、顧客が本当に求めているのは機能一覧ではなく、  
**攻撃全体を一つの流れとして把握し、対応できる統合運用体制**です。

とくに日本の製造業は、本社だけでなく、地方工場、協力会社、部品会社、委託先などに広く分散しています。  
このような環境では、最も強い1社の防御力ではなく、**最も弱い接点の運用レベル**が、サプライチェーン全体のリスクを左右します。

また、地方の中堅・中小製造企業では、  
セキュリティの必要性は認識していても、  
人材確保や、Web・認証・サーバ・対応・フォレンジックを横断して理解できる人材の継続的な確保が難しいのが実情です。

この点で必要なのは、個別製品を個別に導入・運用する構成ではなく、  
**クラウドSaaS型の統合セキュリティ基盤と統合セキュリティ監視運用**によって、  
運用負荷を下げながら、実効性ある対応力を確保できる仕組みです。

# 2. なぜPLURA-XDRなのか

PLURA-XDRは、単なるWeb Application FirewallでもEDRでもありません。  
中核にあるのは、**Web + アカウント + サーバ + 情報持ち出し + 自動対応 + フォレンジック**を一つの流れとして結び付けることです。

## 2-1. 主な差別化ポイント

- **ゼロデイ攻撃を前提とした検知視点**
- **クレデンシャルスタッフィング対策**
- **リクエスト/レスポンス本文ベースの詳細分析**
- **機微情報持ち出しの兆候検知**
- **ランサムウェア対策**
- **APT対応**
- **Web-Server相関分析**
- **自動対応とフォレンジック連携**

つまり、既知の攻撃を防ぐだけではなく、  
**攻撃全体の流れを解釈し、運用できる仕組み**です。

## 2-2. 技術的に何が違うのか

PLURA-XDRの重要な違いは、  
単発イベントを見るのではなく、  
実際の侵害の流れを、より深く一貫して解釈できるように設計されている点にあります。

### リクエスト/レスポンス本文の詳細分析

多くのセキュリティ製品はURL、ヘッダ、イベント概要を中心に見ますが、  
実際の侵害判断には、リクエスト本文やレスポンス本文のレベルでの分析が重要です。

PLURA-XDRはこれにより、

- 既知パターンの検知
- 非定型攻撃の兆候把握
- 認証回避の試みの分析
- 機微情報持ち出し兆候の把握

まで踏み込んで解釈できます。

### Web-Server相関分析

Web攻撃が実際の侵害につながったかを判断するには、  
Webイベントとサーバ挙動を同時に見る必要があります。

PLURA-XDRは、

- Webリクエストと認証フロー
- サーバプロセス、ファイル、ネットワーク挙動
- 横展開や後続の異常挙動

を、一つのインシデントの流れとして分析できます。

つまり、単なる検知ではなく、  
**実際に侵害へ発展したのか**をより早く判断できます。

### 自動対応とフォレンジック連携

多くの組織では、検知後の対応と証跡確保が別運用になっています。

一方PLURA-XDRは、  
検知、アラート、対応、フォレンジック確保を  
一つの運用フローとしてつなげられるよう設計されています。

これは初動対応の迅速化だけでなく、  
事後分析、報告、再発防止の面でも大きな意味を持ちます。

## 2-3. すべての基準はMITRE ATT&CK

PLURA-XDRの特徴は、  
検知項目を機能単位ではなく、  
**MITRE ATT&CKに基づいて解釈・運用する**ことです。

たとえば、

- クレデンシャルスタッフィングは **Initial Access / Credential Access**
- 情報持ち出しは **Collection / Exfiltration**
- ランサムウェアは **Impact**
- APTは **複数のATT&CK段階が連なる攻撃フロー**

として整理できます。

すなわち、

> **PLURA-XDRは、Web攻撃、アカウント攻撃、内部拡散、情報持ち出し、ランサムウェア、APTまでを、MITRE ATT&CKベースで一つの流れとして分析・対応する統合セキュリティプラットフォーム**です。

## 2-4. MythosクラスのAI新脅威への対応力

近年、AIは脆弱性探索や攻撃自動化のスピードを高めています。  
こうした環境では、シグネチャ中心の個別製品だけでは十分ではありません。

PLURA-XDRは、次の構造で差別化します。

- **リクエスト/レスポンス本文の詳細分析**による未知攻撃の兆候把握
- **Web-Server相関分析**によるサーバ側挙動との接続
- **自動対応 + フォレンジック**による検知後対応の一体化

そのため、NS Solutionsが日本の顧客に提案する際にも、  
PLURA-XDRは **AI時代の新脅威にも対応可能な次世代統合セキュリティ基盤**として説明できます。

## 2-5. なぜ日本の製造業・サプライチェーンに適しているのか

日本の製造業やサプライチェーン環境では、  
Webから始まる攻撃が、アカウント侵害、サーバ侵害、重要情報アクセスへと発展し得ます。

したがって必要なのは、  
機能ごとに分断された構成ではなく、  
**Web・アカウント・サーバ・情報持ち出しを一つのインシデントとして把握できるプラットフォーム**です。

そして地方の中堅・中小製造業まで視野に入れるなら、  
複数製品を各社が個別に運用するよりも、  
**クラウドSaaS型の統合基盤 + 統合監視運用**のほうが、はるかに現実的です。

つまりPLURA-XDRは、  
大企業本社だけでなく、地方拠点や協力会社まで含めた  
**サプライチェーン全体に適用できる運用モデル**として説明できます。

# 3. どのように始めるか

本提案の利点は、  
いきなり全面更改を求めないことです。

重要なのは、**小さく始め、早く検証し、段階的に拡大すること**です。

## 3-1. 導入の方向性

PLURA-XDRは次の方向で導入できます。

**1) クラウドSaaS型統合セキュリティ基盤として適用**  
Web、アカウント、サーバ、情報持ち出し、対応、フォレンジックを  
一つの運用体制へ統合

**2) 統合セキュリティ監視運用モデルと組み合わせる**  
検知、分析、対応、報告を一体的に支援する  
マネージドサービス型へ拡張

これにより、顧客は複雑なセキュリティ構成を自ら組み立てるのではなく、  
少人数でも実効的な検知・分析・対応フローを運用できるようになります。

## 3-2. NS Solutionsとの役割分担

- **NS Solutions**: 顧客フロント、インフラ設計、構築、運用、サービス提供
- **PLURA-XDR**: プラットフォーム、検知・分析エンジン、脅威分析技術、対応基盤
- **共同運営**: パイロット、運用基準策定、高度化、共同提案

つまり本提案は、単なる再販ではなく、  
**NS Solutions主導のサービス型セキュリティモデルを共同設計する構想**です。

## 3-3. 段階的な開始方法

- インターネット公開Webサービスを1つ選定
- ログインまたは主要API区間を1つ選定
- 関連サーバへ段階的に適用
- Web-アカウント-サーバの可視性を確認
- 情報持ち出し兆候と対応フローを検証
- コンソール、通知、フォレンジック取得の流れを確認

つまり、最初から全面展開するのではなく、  
**小さく始めて、早く検証し、段階的に拡大するアプローチ**が適しています。

## 3-4. 想定KPI例

| 項目 | 目標例 |
| --- | --- |
| 検知後の一次分析時間 | 従来比 **50%以上短縮** |
| 初動対応時間 | 従来比 **40%以上短縮** |
| コンソール切替回数 | 1インシデントあたり **3以上 → 1レベル** |
| フォレンジック証跡確保率 | 主要イベント基準 **100%確保** |
| 定型反復業務 | 自動化範囲に応じて **50%以上削減期待** |

これらは、運用の複雑さ、対応遅延、人材制約といった課題を、  
具体的な運用改善指標に置き換えて示すものです。

## 3-5. 最終的に目指すモデル

最終的には、  
大企業本社だけでなく、  
地方の中堅・中小協力会社まで含めてカバーできる  
**Managed Security / Cloud Security Serviceモデル**へ発展させることが重要です。

すなわち本提案は、単なる製品販売ではなく、  
**サプライチェーン全体が現実に導入・運用できるサービス型統合セキュリティモデル**を共に構築する提案です。
