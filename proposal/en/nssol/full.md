## 4️⃣ Supply Chain Security: Why Nissan and Similar Cases Matter

Supply chain attacks must not be treated as isolated news events.
They reveal a structural truth.

Attackers rarely choose the strongest target first.
They look for the easiest path into a broader ecosystem.
In a manufacturing and supply chain environment, that path may begin with:

- a web-facing service,
- a supplier-side credential compromise,
- a poorly monitored API,
- or an inadequately protected server in a connected business unit.

Once access is obtained, the attack can move from initial intrusion to persistence, lateral movement, data collection, and destructive action.

This is why the security conversation must change.
The issue is no longer just preventing “one attack.”
The issue is whether the organization can follow and respond to the attack as it moves through multiple stages.

For Japanese manufacturers and their ecosystems, this is especially important because many operations still depend on interconnected environments, long vendor chains, and mixed legacy-modern infrastructure.

So in the proposal, supply chain security should be framed this way:

> A supply chain incident is not evidence that one product failed.
> It is evidence that the operating model was too fragmented to see the whole attack path.

PLURA-XDR addresses this gap by connecting web, authentication, server behavior, and response into one platform.
That makes it more suitable for supply chain environments than a set of disconnected point products.

## 8️⃣ Why PLURA-XDR Fits This Mission

PLURA-XDR is designed around a simple principle.
Attacks should not be viewed as isolated alerts.
They should be understood as connected behavior.

That is why PLURA-XDR combines the following into one operating structure:

- Zero-day attack detection
- Credential stuffing response
- Request and response body analysis
- Sensitive data exfiltration detection
- Web-to-server correlation
- Ransomware and APT response
- Automated response and forensic linkage
- MITRE ATT&CK-based interpretation of attacker behavior

This matters because most customer environments still suffer from fragmentation.
The WAF sees one thing.
Authentication logs show another.
Servers reveal something later.
Response begins only after a human manually pieces events together.

PLURA-XDR reduces that fragmentation.
It is built to follow the attack flow itself.

### 8-1) Web is the start of the story
Many attacks begin in the web layer.
But they do not end there.
PLURA-Web analyzes requests and responses in detail, making it possible to identify not only known attack signatures but also suspicious patterns, session anomalies, and signs of data exposure.

### 8-2) Identity and session behavior matter
Credential attacks are not just login failures.
They are behavioral patterns.
PLURA-XDR can interpret repeated account attempts, session anomalies, and suspicious access flows as part of a larger incident context.

### 8-3) The server is where consequences become real
Once an attacker reaches the server layer, the risk escalates.
PLURA-EDR makes it possible to connect web intrusion, server execution, persistence attempts, lateral movement, LOLBAS-style abuse, and ransomware behavior into one sequence.

### 8-4) Response and forensics must not be separate phases
In many organizations, detection comes first and evidence gathering comes later.
That delay weakens both response quality and post-incident understanding.
PLURA-XDR links detection, automated action, and forensic evidence handling in one operational structure.

This is why PLURA-XDR is a strong fit for the message NS Solutions wants to own.
It does not merely add another alert source.
It helps create a security operating model that can actually be defended in front of customers.

## 1️⃣1️⃣ Before and After: What Changes with NS Solutions + PLURA-XDR

### Scenario A. Credential-based web attack
**Before**
- Login failures are distributed across systems.
- Teams must manually compare IPs, access paths, account logs, and application behavior.
- It is difficult to distinguish automated attack traffic from normal user error quickly.

**After**
- Web requests, sessions, authentication results, and abnormal access patterns are analyzed together.
- Credential stuffing behavior is recognized as an attack flow, not as isolated login failures.
- Response can move quickly from detection to blocking and follow-up analysis.

### Scenario B. Web intrusion leading to server compromise
**Before**
- Web logs and server behavior are handled separately.
- Root cause analysis is slow.
- Post-incident forensics is treated as an extra step.

**After**
- The sequence from web attack to server-side execution is connected in one view.
- Persistence, LOLBAS-style abuse, lateral movement, and ransomware precursors can be traced more clearly.
- Detection, response, and evidence preservation are linked from the start.

### Scenario C. Supply chain exposure across connected organizations
**Before**
- Security maturity varies widely between entities.
- Small incidents are not easily connected to larger attack paths.
- Visibility is fragmented and leadership receives partial answers.

**After**
- The operating model is standardized.
- Incident interpretation becomes more consistent across environments.
- NS Solutions can explain security not as separate products, but as one managed protection framework.
