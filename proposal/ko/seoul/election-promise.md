# 핵심 공약 추가

## 서울시 표준로깅 시스템 도입

### “보안의 출발점은 기록입니다.”

사이버보안에서 가장 중요한 것은 로그입니다.
공격이 발생했을 때 무엇이 들어왔고, 어떤 계정이 사용되었고, 어떤 서버에서 어떤 행위가 일어났으며, 어떤 데이터가 접근·변경·유출되었는지를 확인할 수 있어야 합니다.

그러나 현재 많은 기관의 정보보안 체계는 로그를 충분히 남기지 못하거나, 남기더라도 기관별·시스템별 기준이 달라 공격의 전체 흐름을 재구성하기 어렵습니다.

로그 기준이 부실하면 다음과 같은 문제가 발생합니다.

| 문제               | 결과                          |
| ---------------- | --------------------------- |
| 로그 수집 기준 부재      | 어떤 이벤트를 남겨야 하는지 기관마다 다름     |
| 로그 형식 불일치        | 서울시청, 자치구, 산하기관 간 통합 분석 어려움 |
| 중요 로그 누락         | 공격 원인 분석 실패                 |
| 보존 기간 불명확        | 사고 이후 증거 확보 실패              |
| 개인정보 접근 로그 미흡    | 피해 규모 산정 실패                 |
| 웹 요청·계정·서버 로그 분리 | 공격 흐름 파악 실패                 |
| 수동 보고 중심         | 대응 방안 도출 지연                 |

결과적으로 사고가 발생해도 **공격 원인 분석, 피해 규모 산정, 재발 방지 대책 수립, 책임 소재 확인, 시민 안내**가 모두 늦어질 수밖에 없습니다.

따라서 서울시는 통합 정보보안 플랫폼 구축에 앞서, 그 기반이 되는 **서울시 표준로깅 시스템**을 도입해야 합니다.

# 서울시 표준로깅 시스템의 목표

## “서울시 전체 보안 데이터를 하나의 기준으로 기록하겠습니다.”

서울시 표준로깅 시스템은 서울시청, 25개 자치구, 산하기관, 투자·출연기관, 시립병원, 복지시설의 보안 로그를 동일한 기준으로 수집·보존·분석하기 위한 공통 체계입니다.

단순히 로그를 많이 모으는 것이 목적이 아닙니다.
**사고가 발생했을 때 공격 흐름을 재구성할 수 있는 로그**를 남기는 것이 핵심입니다.

# 표준로깅 시스템의 핵심 원칙

## 1. 공격 흐름을 재구성할 수 있어야 합니다

단일 로그만으로는 공격을 이해할 수 없습니다.
웹 요청, 계정 행위, 서버 이벤트, 개인정보 접근 기록이 하나의 흐름으로 연결되어야 합니다.

예를 들어 다음과 같은 분석이 가능해야 합니다.

```text
외부 공격 요청
→ 로그인 시도
→ 계정 탈취
→ 관리자 권한 상승
→ 서버 명령 실행
→ 내부 이동
→ 개인정보 조회
→ 외부 전송
→ 차단 및 복구
```

이 흐름을 확인하지 못하면 사고 대응은 추측에 의존하게 됩니다.

서울시청은 A 방식, 자치구는 B 방식, 산하기관은 C 방식으로 로그를 남기면 통합 분석이 불가능합니다.

따라서 서울시는 다음 기준을 통일해야 합니다.

| 로그 형식   | 기관 간 비교 가능한 공통 포맷 적용 |
| 로그 시간   | 표준 시간 동기화            |
| 로그 보존   | 중요도별 보존 기간 기준 마련     |
| 로그 무결성  | 위변조 방지 및 원본 보존       |
| 로그 접근권한 | 열람·수정·삭제 권한 통제       |
| 로그 검색   | 사고 시 즉시 검색 가능한 구조    |
| 로그 연계   | 통합 정보보안 플랫폼과 자동 연동   |

## 3. 피해 규모를 산정할 수 있어야 합니다

보안사고 이후 가장 중요한 질문은 다음입니다.

```text
누가 공격했는가?
어디로 들어왔는가?
어떤 계정이 사용되었는가?
어떤 시스템이 영향을 받았는가?
어떤 개인정보가 조회되었는가?
얼마나 많은 시민이 영향을 받았는가?
현재도 공격이 진행 중인가?
어떤 조치를 해야 재발을 막을 수 있는가?
```

이 질문에 답하려면 표준화된 로그가 있어야 합니다.
로그가 없으면 피해 규모도, 시민 안내도, 재발 방지도 제대로 할 수 없습니다.

# 통합 정보보안 플랫폼과의 관계

## “표준로깅 시스템이 있어야 통합 플랫폼이 완성됩니다.”

서울시 통합 정보보안 플랫폼은 표준로깅 시스템 위에서 완성됩니다.

로그 기준이 없으면 통합 플랫폼은 단순히 여러 장비의 알림을 모아 보는 수준에 머물게 됩니다.
반대로 표준로깅 시스템이 구축되면 서울시 전체의 보안 상황을 하나의 기준으로 분석할 수 있습니다.

```text
서울시 표준로깅 시스템
→ 서울시청·자치구·산하기관 로그 표준화
→ 통합 정보보안 플랫폼 연계
→ 공격 흐름 분석
→ 피해 규모 산정
→ 등급별 대응
→ 선조치·후보고
→ 재발 방지
→ 시민 신뢰 회복
```

따라서 공약의 구조는 다음처럼 잡는 것이 좋습니다.

```text
1단계: 서울시 표준로깅 시스템 도입
2단계: 통합 정보보안 플랫폼 구축
3단계: 미토스급 해킹 등급별 대응 체계 운영
4단계: 선조치·후보고 제도화
5단계: 분기별 점검과 공개 토론으로 제도 정착
```

# 기존 공약에 반영한 수정 구조

## 서울시 사이버보안 8대 공약

**1. 서울시 표준로깅 시스템 도입**
서울시청, 자치구, 산하기관의 보안 로그를 하나의 기준으로 수집·보존·분석하겠습니다.

**2. 서울 Cyber One Platform 구축**
표준화된 로그를 기반으로 서울시 전체 통합 정보보안 플랫폼을 구축하겠습니다.

**3. 레거시 정보보안 체계 제거**
기관별 개별 장비, 수동 보고, 중복 관제 중심의 낡은 보안체계를 단계적으로 제거하겠습니다.

**4. 미토스급 해킹 등급별 대응 체계 구축**
AI 기반 고속 공격에 맞춰 긴급·심각·경계·주의·관심 등급별 대응 기준을 마련하겠습니다.

**5. 긴급 상황 선조치·후보고 제도화**
시민 피해를 막기 위해 긴급 상황에서는 차단, 격리, 계정 잠금, 임시 중단을 먼저 할 수 있도록 하겠습니다.

**6. 사이버보안 통합 정책 담당관 지정**
정책과 실무를 분리하고, 서울시 전체 사이버보안 정책을 총괄하는 전문 담당관 체계를 만들겠습니다.

**7. 소프트웨어 구성명세서와 상시 패치 의무화**
서울시에 납품되는 소프트웨어의 구성과 취약점을 관리하고, 운영 중에도 상시 패치를 의무화하겠습니다.

**8. 분기별 점검과 공개 토론 운영**
새로운 제도가 현장에 정착되도록 분기별 모니터링, 점검, 공개 토론을 운영하겠습니다.

# 공약집 본문 삽입용 문안

## 서울시 표준로깅 시스템 도입

사이버보안의 출발점은 기록입니다.

공격이 발생했을 때 정확한 로그가 없다면 원인 분석도, 피해 규모 산정도, 재발 방지 대책도 불가능합니다. 그동안 많은 공공기관은 보안장비와 관제체계를 운영해 왔지만, 정작 가장 중요한 로그에 대해서는 기관별 기준이 다르고, 어떤 로그를 반드시 남겨야 하는지에 대한 실질적인 지침이 부족했습니다.

그 결과 사고가 발생하면 다음과 같은 문제가 반복됩니다.

```text
공격이 어디서 시작되었는지 모른다.
어떤 계정이 탈취되었는지 모른다.
어떤 서버가 영향을 받았는지 모른다.
어떤 개인정보가 조회되었는지 모른다.
피해 규모를 정확히 산정하지 못한다.
대응 방안을 신속하게 도출하지 못한다.
재발 방지 대책이 추상적으로 끝난다.
```

서울시는 이 문제를 근본적으로 해결하겠습니다.

서울시청, 25개 자치구, 산하기관, 투자·출연기관, 시립병원, 복지시설에 적용되는 **서울시 표준로깅 시스템**을 도입하겠습니다. 웹 요청, 계정 행위, 서버 이벤트, 개인정보 접근 기록, 관리자 작업, 패치 이력, 사고 대응 조치를 하나의 기준으로 기록하고 분석하겠습니다.

표준로깅 시스템은 서울시 통합 정보보안 플랫폼의 기반입니다.
표준화된 로그가 있어야 서울시 전체의 보안 상황을 하나의 화면에서 보고, 하나의 기준으로 판단하고, 하나의 절차로 대응할 수 있습니다.

서울시 표준로깅 시스템을 통해 다음을 가능하게 하겠습니다.

* 공격 원인 분석
* 피해 규모 산정
* 개인정보 유출 여부 확인
* 미토스급 AI 해킹 흐름 분석
* 기관별 대응 이력 관리
* 선조치·후보고 검증
* 상시 패치 이행 확인
* 재발 방지 대책 수립

서울시 전체 통합 보안 플랫폼은 표준로깅 시스템 위에서 완성됩니다.
서울의 보안은 더 이상 장비를 많이 사는 방식이 아니라, **무엇을 기록하고, 어떻게 분석하고, 얼마나 빠르게 대응하느냐**로 평가받아야 합니다.
