# 1. 왜 지금 서울시 사이버안보 대전환인가

서울은 더 안전하고 성숙한 스마트시티로 발전해야 합니다.  
스마트시티의 핵심은 편리함만이 아닙니다. 시민의 개인정보가 안전하고, 행정서비스가 멈추지 않으며, 위기 상황에서도 도시 기능이 유지되는 것입니다.

서울의 행정, 교통, 복지, 의료, 상수도, 재난안전, 시민 민원 서비스는 이미 디지털 시스템 위에서 운영되고 있습니다. 따라서 사이버공격은 더 이상 단순한 전산 장애가 아닙니다. 시민의 개인정보, 행정서비스, 도시 기능을 위협하는 새로운 안전사고이자 도시 재난입니다.

특히 AI 공격은 이미 현실화되고 있습니다.  
취약점 탐지, 공격 경로 분석, 악성코드 변형, 계정 탈취, 자동화된 침투 시도가 과거보다 훨씬 빠른 속도로 이루어질 수 있습니다. 미토스급 AI 해킹은 기존의 사후 보고, 개별 장비, 기관별 대응 중심의 보안체계로는 대응하기 어렵습니다.

이제 서울시는 사이버안보 분야의 근본적인 전환을 추진해야 합니다.

그 출발점은 **서울시 표준로깅 시스템**입니다.  
로그가 없으면 공격 원인도 알 수 없고, 피해 규모도 산정할 수 없으며, 시민에게 어떤 정보가 영향을 받았는지도 설명할 수 없습니다. 지금까지 정보보안은 장비 도입과 관제 확대에 집중해 왔지만, 정작 가장 중요한 로그에 대해서는 제대로 된 기준과 지침이 부족했습니다.

서울시는 이제 다음 단계로 나아가야 합니다.

**서울시청, 25개 자치구, 산하기관, 투자·출연기관, 시립병원, 복지시설의 보안 기록을 하나의 기준으로 통일하고, 이를 기반으로 서울시 전체 통합 정보보안 플랫폼을 구축하겠습니다.**

이 플랫폼은 단순한 보안장비의 집합이 아닙니다.  
서울 전체의 로그, 위협, 취약점, 패치, 사고 대응, 선조치·후보고 이력을 하나의 기준으로 관리하는 **서울형 사이버안보 운영체계**입니다.

이것은 단순한 기술 통합이 아니라 **책임 행정의 기반**입니다.  
사이버사고가 발생했을 때 시민은 “어디에서 시작됐는지”, “어떤 피해가 있었는지”, “누가 어떤 조치를 했는지”, “앞으로 어떻게 막을 것인지”를 요구합니다. 서울시청, 자치구, 산하기관이 서로 다른 기준으로 기록하고 대응하면 시민에게 책임 있게 설명할 수 없습니다. 서울시 전체가 같은 기준으로 기록하고, 같은 플랫폼에서 분석하고, 같은 절차로 대응해야 합니다.

서울을 더 성숙한 스마트시티로 발전시키기 위해서는 안전사고 없는 디지털 기반이 필요합니다. 그리고 그 기반은 **표준화된 로그, 통합된 플랫폼, 등급별 대응 체계, 긴급 시 선조치·후보고가 가능한 제도**에서 시작됩니다.

**기록하고, 분석하고, 즉시 대응하는 서울.**  
**AI 공격에도 흔들리지 않는 안전한 스마트시티 서울.**  
그 변화를 시작하겠습니다.

# 4. 핵심 공약 1: 서울시 표준로깅 시스템 도입

## “보안의 출발점은 기록입니다.”

사이버보안에서 가장 중요한 것은 로그입니다.  
공격이 발생했을 때 무엇이 들어왔고, 어떤 계정이 사용되었고, 어떤 서버에서 어떤 행위가 일어났으며, 어떤 데이터가 접근·변경·유출되었는지를 확인할 수 있어야 합니다.

그러나 현재 많은 기관의 정보보안 체계는 로그를 충분히 남기지 못하거나, 남기더라도 기관별·시스템별 기준이 달라 공격의 전체 흐름을 재구성하기 어렵습니다.

로그 기준이 부실하면 다음과 같은 문제가 발생합니다.

| 문제 | 결과 |
|---|---|
| 로그 수집 기준 부재 | 어떤 이벤트를 남겨야 하는지 기관마다 다름 |
| 로그 형식 불일치 | 서울시청, 자치구, 산하기관 간 통합 분석 어려움 |
| 중요 로그 누락 | 공격 원인 분석 실패 |
| 보존 기간 불명확 | 사고 이후 증거 확보 실패 |
| 개인정보 접근 로그 미흡 | 피해 규모 산정 실패 |
| 웹 요청·계정·서버 로그 분리 | 공격 흐름 파악 실패 |
| 수동 보고 중심 | 대응 방안 도출 지연 |

결과적으로 사고가 발생해도 **공격 원인 분석, 피해 규모 산정, 재발 방지 대책 수립, 책임 소재 확인, 시민 안내**가 모두 늦어질 수밖에 없습니다.

따라서 서울시는 통합 정보보안 플랫폼 구축에 앞서, 그 기반이 되는 **서울시 표준로깅 시스템**을 도입해야 합니다.

## “서울시 전체 보안 데이터를 하나의 기준으로 기록하겠습니다.”

서울시 표준로깅 시스템은 서울시청, 25개 자치구, 산하기관, 투자·출연기관, 시립병원, 복지시설의 보안 로그를 동일한 기준으로 수집·보존·분석하기 위한 공통 체계입니다.

단순히 로그를 많이 모으는 것이 목적이 아닙니다.  
**사고가 발생했을 때 공격 흐름을 재구성할 수 있는 로그**를 남기는 것이 핵심입니다.

# 7. 표준로깅 시스템의 핵심 원칙

## 1. 공격 흐름을 재구성할 수 있어야 합니다

단일 로그만으로는 공격을 이해할 수 없습니다.  
웹 요청, 계정 행위, 서버 이벤트, 개인정보 접근 기록이 하나의 흐름으로 연결되어야 합니다.

예를 들어 다음과 같은 분석이 가능해야 합니다.

```text
외부 공격 요청
→ 로그인 시도
→ 계정 탈취
→ 관리자 권한 상승
→ 서버 명령 실행
→ 내부 이동
→ 개인정보 조회
→ 외부 전송
→ 차단 및 복구
```

이 흐름을 확인하지 못하면 사고 대응은 추측에 의존하게 됩니다.

## 2. 서울시 전체가 같은 로그 기준을 사용해야 합니다

서울시청은 A 방식, 자치구는 B 방식, 산하기관은 C 방식으로 로그를 남기면 통합 분석이 불가능합니다.

따라서 서울시는 다음 기준을 통일해야 합니다.

| 항목 | 표준화 내용 |
|---|---|
| 로그 항목 | 반드시 남겨야 할 필수 로그 정의 |
| 로그 형식 | 기관 간 비교 가능한 공통 포맷 적용 |
| 로그 시간 | 표준 시간 동기화 |
| 로그 보존 | 중요도별 보존 기간 기준 마련 |
| 로그 무결성 | 위변조 방지 및 원본 보존 |
| 로그 접근권한 | 열람·수정·삭제 권한 통제 |
| 로그 검색 | 사고 시 즉시 검색 가능한 구조 |
| 로그 연계 | 통합 정보보안 플랫폼과 자동 연동 |

## 3. 피해 규모를 산정할 수 있어야 합니다

보안사고 이후 가장 중요한 질문은 다음입니다.

```text
누가 공격했는가?
어디로 들어왔는가?
어떤 계정이 사용되었는가?
어떤 시스템이 영향을 받았는가?
어떤 개인정보가 조회되었는가?
얼마나 많은 시민이 영향을 받았는가?
현재도 공격이 진행 중인가?
어떤 조치를 해야 재발을 막을 수 있는가?
```

이 질문에 답하려면 표준화된 로그가 있어야 합니다.  
로그가 없으면 피해 규모도, 시민 안내도, 재발 방지도 제대로 할 수 없습니다.

# 8. 통합 정보보안 플랫폼과 표준로깅 시스템의 관계

## “표준로깅 시스템이 있어야 통합 플랫폼이 완성됩니다.”

서울시 통합 정보보안 플랫폼은 표준로깅 시스템 위에서 완성됩니다.

로그 기준이 없으면 통합 플랫폼은 단순히 여러 장비의 알림을 모아 보는 수준에 머물게 됩니다.  
반대로 표준로깅 시스템이 구축되면 서울시 전체의 보안 상황을 하나의 기준으로 분석할 수 있습니다.

```text
서울시 표준로깅 시스템
→ 서울시청·자치구·산하기관 로그 표준화
→ 통합 정보보안 플랫폼 연계
→ 공격 흐름 분석
→ 피해 규모 산정
→ 등급별 대응
→ 선조치·후보고
→ 재발 방지
→ 시민 신뢰 회복
```

따라서 공약의 구조는 다음처럼 추진합니다.

```text
1단계: 서울시 표준로깅 시스템 도입
2단계: 통합 정보보안 플랫폼 구축
3단계: 미토스급 해킹 등급별 대응 체계 운영
4단계: 선조치·후보고 제도화
5단계: 분기별 점검과 공개 토론으로 제도 정착
```

# 9. 핵심 공약 2: 서울시 통합 정보보안 플랫폼 구축

## “서울시청, 자치구, 산하기관이 하나의 기준으로 협의하고 대응하겠습니다.”

서울시청, 25개 자치구, 산하기관, 투자·출연기관, 시립병원, 복지시설이 각각 따로 움직이는 보안체계로는 미토스급 AI 해킹에 대응할 수 없습니다.

서울시는 표준로깅 시스템을 기반으로 **서울시 통합 정보보안 플랫폼**을 구축하겠습니다.

이 플랫폼은 단순한 기술 통합이 아닙니다.  
**책임 행정을 위한 통합 기반**입니다. 사이버사고가 발생했을 때 원인, 피해 범위, 조치 담당, 조치 시각, 재발 방지 대책을 시민에게 설명할 수 있어야 합니다. 이를 위해 서울시청, 자치구, 산하기관은 같은 기준으로 기록하고, 같은 플랫폼에서 분석하고, 같은 절차로 대응해야 합니다.

이 플랫폼은 다음을 통합합니다.

| 구분 | 내용 |
|---|---|
| 통합 관제 | 서울시청·자치구·산하기관 보안 이벤트 통합 수집 |
| 통합 분석 | 웹 요청, 시스템 이벤트, 계정 행위, 서버 로그, 개인정보 접근 기록 연계 분석 |
| AI 대응 | 미토스급 자동화 공격 징후 분석 및 대응 우선순위 판단 |
| MITRE ATT&CK 매핑 | 공격 흐름을 전술·기술·절차 기준으로 분류하고 대응 수준 점검 |
| 웹 공격 대응 | 웹방화벽, 웹 제로데이 탐지, 크리덴셜 스터핑 대응 |
| 자동 대응 | 악성 IP 차단, 계정 잠금, 서버 격리, 위험 세션 차단 |
| 패치 관리 | 기관별 보안 패치 현황, 지연 현황, 조치 이력 관리 |
| 사고 관리 | 선조치, 후보고, 재발 방지, 책임기관 지정 |
| 정책 관리 | 서울시 전체 보안 기준, 예외 승인, 점검 결과 통합 관리 |
| 공개 점검 | 분기별 이행 현황 점검 및 공개 토론 운영 |

## 통합 플랫폼 자체 보안

서울시 통합 정보보안 플랫폼은 중요한 보안 데이터를 다루는 만큼, 그 자체가 새로운 공격 대상이 되지 않도록 별도의 보호체계를 적용하겠습니다.

| 보호 항목 | 내용 |
|---|---|
| 데이터 암호화 | 저장·전송 구간 암호화 |
| 접근권한 통제 | 역할 기반 접근권한, 최소권한 원칙 적용 |
| 관리자 행위 감사 | 조회·검색·반출·정책 변경 이력 기록 |
| 로그 무결성 | 원본 로그 위변조 방지, 보존 정책 적용 |

| 외부 반출 통제 | 민감 로그 다운로드·반출 승인제 |
| 정기 검증 | 플랫폼 자체 취약점 점검과 침해 대응 훈련 |

# 10. 핵심 공약 3: 레거시 정보보안 체계 제거

## “인증 항목만 채우는 보안에서 실제 공격을 막는 보안으로 전환하겠습니다.”

서울시의 정보보안은 더 이상 기관별 장비 도입, 개별 관제, 수동 보고, 사후 점검 방식에 머물러서는 안 됩니다.

서울시는 레거시 정보보안 제품과 절차를 평가하여 불필요하고 의미 없이 예산을 낭비하는 요소를 제거하겠습니다.  
특히 **인증 항목을 충족한다는 이유만으로 사용되고 있지만 실제 공격 대응에 기여하지 못하는 제품과 절차는 과감히 걷어내겠습니다.**

## 평가 기준

| 평가 항목 | 내용 |
|---|---|
| 실제 탐지 효과 | 최신 공격과 미토스급 자동화 공격에 대응 가능한가 |
| 로그 연계성 | 서울시 표준로깅 시스템과 연계 가능한가 |
| 통합 플랫폼 연계성 | 통합 정보보안 플랫폼에서 분석·대응 가능한가 |
| 자동 대응 가능성 | 차단, 격리, 계정 잠금 등 대응 자동화에 기여하는가 |
| 운영 부담 | 실무자에게 과도한 수동 운영 부담을 주지 않는가 |
| 중복 여부 | 다른 시스템과 기능이 중복되지 않는가 |
| 인증 의존성 | 인증 항목 충족 외에 실질적 보안 가치가 있는가 |
| 비용 대비 효과 | 예산 대비 보안 효과가 명확한가 |

## 추진 원칙

1. **신규 레거시 보안장비 도입 제한**  
   통합 플랫폼과 연계되지 않는 개별 장비 도입을 제한합니다.

2. **기존 보안장비 효과성 평가**  
   기존 장비는 즉시 폐기하는 것이 아니라, 실제 탐지·분석·대응 효과를 평가합니다.

3. **인증 항목 목적 제품 정리**  
   인증 항목을 채우기 위한 목적으로만 유지되는 제품은 통합·대체·폐지합니다.

4. **기존 장비 단계적 통합**  
   필요한 장비는 로그·정책·대응 절차를 통합 플랫폼으로 흡수합니다.

5. **유예 기간과 전환 인센티브 부여**  
   자치구와 산하기관의 예산·계약·폐쇄망 환경을 고려해 합리적 유예 기간을 두고, 우수 전환 기관에는 예산·평가 인센티브를 제공합니다.

6. **수동 보고 축소**  
   사고 보고, 패치 현황, 조치 이력은 플랫폼에서 자동 관리되도록 전환합니다.

7. **서울시 표준 보안 운영 절차 적용**  
   자치구와 산하기관도 동일한 대응 기준을 사용하도록 합니다.

# 11. 핵심 공약 4: 미토스급 해킹 등급별 대응 체계 구축

## “공격 속도에 맞는 행정 대응 체계를 만들겠습니다.”

미토스급 AI 해킹은 사람이 회의하고 보고서를 올리는 속도보다 빠릅니다.  
따라서 서울시는 사이버 위협을 등급화하고, 등급별 대응 권한과 절차를 명확히 하겠습니다.

공약에서 말하는 **미토스급 AI 해킹**은 단순히 “AI가 만든 공격”이라는 뜻이 아닙니다.  
다음과 같은 고속·자동화·연쇄형 공격을 의미합니다.

| 유형 | 설명 |
|---|---|
| 제로데이 취약점 연쇄 공격 | 공개되지 않은 취약점 또는 패치 전 취약점을 빠르게 탐색·악용 |
| 웹·API 취약점 자동 탐색 | 대민 서비스, 민원 시스템, API의 취약점을 자동 탐색 |
| 자동화된 크리덴셜 스터핑 | 유출 계정 정보를 이용해 대량 로그인 시도 |
| 계정 탈취 후 내부 이동 | 정상 계정처럼 접속한 뒤 권한 상승과 내부 확산 |
| 랜섬웨어·데이터 유출 결합 | 침투 후 암호화, 협박, 정보 유출을 연계 |
| 방어 우회형 변종 공격 | 탐지 규칙을 피하기 위해 공격 패턴을 빠르게 변형 |

서울시는 이러한 공격을 **MITRE ATT&CK 프레임워크**에 따라 전술·기술·절차 단위로 분류하고, 표준 로그와 통합 플랫폼을 통해 공격 흐름을 재구성하겠습니다.

| 등급 | 상황 | 대응 원칙 |
|---|---|---|
| 긴급 | 대규모 개인정보 유출, 랜섬웨어 확산, 행정서비스 마비, AI 기반 자동화 공격 진행 | 선조치·후보고 |
| 심각 | 주요 시스템 침투 정황, 관리자 계정 탈취, 다수 기관 동시 공격 | 즉시 차단 및 시장 보고 |
| 경계 | 취약점 악용 시도, 비정상 로그인 급증, 악성 트래픽 증가 | 기관별 긴급 점검 및 패치 |
| 주의 | 신규 취약점 공개, 공격 가능성 확인 | 상시 모니터링 및 예방 조치 |
| 관심 | 위협 정보 수집 단계 | 관계기관 공유 및 대비 |

# 12. 핵심 공약 5: 긴급 상황 선조치·후보고 제도화

## “시민 피해를 막기 위해 필요한 조치는 먼저 하겠습니다.”

기존 행정 절차는 보고와 승인에 시간이 걸립니다.  
그러나 사이버공격은 기다려주지 않습니다.

서울시는 긴급 사이버 위협 상황에서 **간소화된 대응 절차**를 도입하고, 필요한 경우 **선조치·후보고**가 가능하도록 제도화하겠습니다.

## 선조치 가능 항목

| 선조치 항목 | 내용 |
|---|---|
| 악성 IP 차단 | 공격 출발지 즉시 차단 |
| 위험 계정 잠금 | 탈취 의심 계정 임시 정지 |
| 서버 격리 | 감염 의심 서버 네트워크 격리 |
| 위험 세션 차단 | 공격 세션 즉시 종료 |
| 취약 서비스 임시 중단 | 시민 피해 확산 방지를 위한 제한적 중단 |
| 긴급 패치 적용 | 공식 승인 전 임시 보완조치 허용 |
| 로그 보존 명령 | 사고 분석을 위한 원본 로그 보존 |
| 유관기관 긴급 전파 | 자치구·산하기관 즉시 공유 |

## 후보고 기준

| 구분 | 기준 |
|---|---|
| 1차 보고 | 조치 후 24시간 이내 |
| 상세 보고 | 조치 후 72시간 이내 |
| 재발 방지 보고 | 7일 이내 |
| 시민 영향 보고 | 개인정보 또는 서비스 영향 발생 시 별도 공개 |

정해진 절차와 권한 안에서 시민 피해를 막기 위해 긴급 조치를 한 담당자는 사후 문책보다 **사후 검증과 제도 개선**을 우선하겠습니다.

# 13. 핵심 공약 6: 사이버보안 실무·기술 담당관 지정과 상설 자문위원 협의체 운영

## “순환보직으로는 AI 해킹에 대응할 수 없습니다.”

서울시에는 이미 정보보안 정책을 담당하는 조직과 담당자가 있습니다.  
문제는 정책 담당자와 실무 담당자가 2년 안팎의 순환보직으로 이동하면서 전문성이 축적되기 어렵다는 점입니다.

AI 해킹 대응은 단순 행정 업무가 아닙니다.  
웹 공격, 제로데이 취약점, 크리덴셜 스터핑, 랜섬웨어, 클라우드 보안, 로그 분석, 포렌식, 개인정보 보호를 이해하고 지속적으로 운영해야 하는 고도의 실무·기술 영역입니다.

서울시는 **사이버보안 실무·기술 담당관**을 지정하고, 최소 5년 이상 같은 분야에서 근무하도록 제도화하겠습니다.

# 14. 핵심 공약 7: MITRE ATT&CK 기반 운영체계, 웹 공격 대응 강화, 상시 패치 의무화

## “서울시 사이버안보를 글로벌 표준과 실제 공격 흐름에 맞추겠습니다.”

서울시 사이버안보 체계는 개별 장비 도입이나 인증 항목 충족에 머물러서는 안 됩니다.  
공격자가 실제로 사용하는 전술·기술·절차를 기준으로 방어체계를 설계해야 합니다.

서울시는 글로벌 표준인 **MITRE ATT&CK 프레임워크**를 기준으로 공격 흐름을 분석하고, 탐지·차단·포렌식·재발 방지 체계를 정비하겠습니다.

## 웹 공격 대응 강화

서울시의 대민 행정서비스는 대부분 웹, 모바일앱, API를 통해 시민과 연결됩니다.  
따라서 웹 공격 대응은 서울시 사이버안보의 최전선입니다.

서울시는 웹방화벽을 단순 차단 장비가 아니라 표준로깅, AI 분석, 통합 대응과 연결된 핵심 방어체계로 강화하겠습니다.

| 대응 분야 | 강화 내용 |
|---|---|
| 웹방화벽 강화 | 대민 서비스 웹 요청·응답 흐름 분석, 정책 통합, 차단 이력 관리 |
| 웹 제로데이 대응 | 알려진 규칙이 없는 이상 요청, 비정상 응답, 공격 흐름 탐지 |
| 크리덴셜 스터핑 대응 | 로그인 요청 본문, 실패·성공 패턴, 세션 행위, 대량 자동화 시도 분석 |
| API 보안 | 비인가 API 호출, 과도한 요청, 권한 우회, 데이터 대량 조회 탐지 |
| 봇·자동화 공격 대응 | 비정상 속도, 반복 시도, 세션 재사용, 자동화 도구 징후 분석 |
| 개인정보 접근 보호 | 대량 조회·다운로드·외부 전송 이상행위 탐지 |
| MITRE ATT&CK 매핑 | 웹 침투 이후 계정 탈취, 권한 상승, 내부 이동까지 공격 흐름 연결 |

## 소프트웨어 구성명세서와 상시 패치 의무화

서울시가 사용하는 소프트웨어와 시스템은 외부 업체가 개발·납품·운영하는 경우가 많습니다.  
따라서 공급망 보안 없이는 서울시 보안도 완성될 수 없습니다.

서울시 주요 정보화 사업에는 다음을 의무화하겠습니다.

| 항목 | 내용 |
|---|---|

| AI 대응 체계 연계 | 신규 취약점과 AI 기반 공격 가능성을 통합 플랫폼에서 관리 |
| 계약 조건 반영 | 패치 지연, 보안 결함 방치 시 책임 기준 명확화 |

## 납품 전 보안 검증의 전환

납품 전 일회성 모의해킹 중심의 검증은 지양하겠습니다.  
대신 개발·납품·운영 전 과정을 서울시 AI 대응 체계와 연계하여 상시 검증하겠습니다.

|---|---|

# 15. 핵심 공약 8: 분기별 모니터링·점검·공개 토론 운영

좋은 정책도 현장에서 작동하지 않으면 의미가 없습니다.  
서울시는 사이버보안 정책이 선언에 그치지 않도록 분기별 점검 체계를 만들겠습니다.

# 16. 후보 메시지

존경하는 서울시민 여러분,

서울은 더 안전하고 성숙한 스마트시티로 발전해야 합니다.  
그 핵심은 편리함을 넘어, 시민의 개인정보와 행정서비스를 안전하게 지키는 것입니다.

AI 공격은 이미 현실화되고 있습니다.  
미토스급 AI 해킹은 제로데이 취약점 탐색, 웹·API 공격, 자동화된 크리덴셜 스터핑, 계정 탈취, 내부 이동을 과거보다 훨씬 빠르게 만들고 있습니다. 기존의 사후 보고 중심 보안체계로는 대응이 어렵습니다.

저는 서울시 사이버안보 체계를 근본적으로 전환하겠습니다.

첫째, **서울시 표준로깅 시스템**을 도입하겠습니다.  
로그가 없으면 공격 원인도, 피해 규모도, 대응 방안도 확인할 수 없습니다. 서울시청, 자치구, 산하기관의 보안 로그를 하나의 기준으로 통일하겠습니다.

둘째, **서울시 통합 정보보안 플랫폼**을 구축하겠습니다.  
이것은 단순한 시스템 통합이 아니라 책임 행정의 기반입니다. 사고가 발생했을 때 원인, 피해, 조치, 재발 방지 대책을 시민에게 설명할 수 있도록 하겠습니다.

셋째, **MITRE ATT&CK 기반 미토스급 AI 해킹 대응 프로세스**를 마련하겠습니다.  
공격 흐름을 글로벌 표준에 맞춰 해석하고, 긴급 상황에서는 시민 피해를 막기 위해 선조치·후보고가 가능하도록 제도화하겠습니다.

넷째, **웹 공격 대응을 강화하겠습니다.**  
대민 행정서비스의 최전선인 웹·앱·API를 보호하기 위해 웹방화벽을 강화하고, 웹 제로데이 공격과 크리덴셜 스터핑에 대응하겠습니다.

다섯째, **사이버보안 실무·기술 담당관을 지정하겠습니다.**  
정책 담당자는 이미 있습니다. 이제는 장기간 전문성을 유지하는 실무·기술 담당관이 필요합니다. 최소 5년 이상 전문성을 축적하고, 실무 보안 전문가 중심의 상설 자문위원 협의체로 협의와 견제가 작동하도록 하겠습니다.

여섯째, **레거시 정보보안을 평가하고 정리하겠습니다.**  
인증 항목을 채우기 위해 실효성 없이 유지되는 제품과 절차는 과감히 걷어내고, 실제 AI 해킹 대응 역량에 예산과 인력을 집중하겠습니다.

서울의 사이버안보는 더 이상 기관별로 따로 움직여서는 안 됩니다.  
서울시청, 자치구, 산하기관이 하나의 기준으로 기록하고, 하나의 플랫폼으로 분석하고, 하나의 체계로 대응해야 합니다.

저는 AI 해킹 대응을 준비한 시장 후보로서, 서울시 사이버안보를 책임 행정의 관점에서 다시 설계하겠습니다.

**기록하고, 분석하고, 즉시 대응하는 서울.**  
**시민에게 설명할 수 있는 책임 행정으로 AI 공격에도 흔들리지 않는 안전한 스마트시티 서울을 만들겠습니다.**

# 17. 선거공보용 짧은 문구

**보안의 출발점은 로그입니다. 서울시 표준로깅 시스템을 도입해 서울시청·자치구·산하기관의 보안 기록을 하나로 통일하고, 이를 기반으로 서울시 통합 정보보안 플랫폼을 완성하겠습니다.**

**로그가 없으면 보안도 없습니다. 서울시 표준로깅 시스템으로 공격 원인, 피해 규모, 대응 방안을 즉시 확인할 수 있는 통합 보안 도시를 만들겠습니다.**

**서울시청·자치구·산하기관의 보안을 하나로 통합하겠습니다. 보안 통합은 시민에게 설명할 수 있는 책임 행정의 출발점입니다.**  

**MITRE ATT&CK 기반으로 공격 흐름을 분석하고, 웹방화벽 강화·웹 제로데이 대응·크리덴셜 스터핑 대응으로 글로벌 최고 수준의 서울형 사이버안보 체계를 만들겠습니다.**

# 부록. 외부 검토 의견 반영 사항

## 1. “미토스급 AI 해킹” 용어 구체화

미토스급 AI 해킹은 대중적 경각심을 주는 정치적 표현이지만, 정책 문서에서는 기술적 정의가 필요합니다.  
따라서 본 공약에서는 이를 다음과 같이 정의합니다.

```text
제로데이 취약점 연쇄 공격
웹·API 취약점 자동 탐색
자동화된 크리덴셜 스터핑
계정 탈취 후 내부 이동
랜섬웨어와 데이터 유출 결합
방어 우회형 변종 공격
```

## 2. 통합 플랫폼 자체 보안 강화

서울시 통합 정보보안 플랫폼은 강력한 방어 체계이지만, 동시에 중요한 로그와 사고 정보를 다루는 핵심 자산입니다.  
따라서 플랫폼 자체에 암호화, 접근권한 통제, 관리자 행위 감사, 로그 무결성, 망·권한 분리, 외부 반출 통제를 적용합니다.

## 3. 레거시 제거의 현실성 확보

레거시 보안 제품은 즉시 폐기하는 방식이 아니라 효과성 평가를 거쳐 통합·대체·폐지합니다.  
자치구와 산하기관의 예산, 계약, 폐쇄망 특성을 고려해 유예 기간을 두고, 우수 전환 기관에는 인센티브를 제공합니다.

## 4. 전문 인력과 예산 확보

AI 해킹 대응은 단기 순환보직만으로는 어렵습니다.  
서울시는 5년 이상 근무하는 사이버보안 실무·기술 담당관을 지정하고, 민간 실무 전문가 중심의 상설 자문위원 협의체를 통해 전문성을 보완합니다. 예산은 중복 장비와 실효성 낮은 레거시 제품을 정리해 통합 플랫폼, 표준로깅, 웹 공격 대응, 상시 패치 체계에 재배분합니다.
