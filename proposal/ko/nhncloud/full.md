# PLURA-XDR 제안서
## NHN Cloud 보안 서비스 통합 경쟁력 강화 제안

## 0️⃣ 제안 요약

NHN Cloud는 App Security Check, Server Security Check, Web Firewall, DDoS Guard, SIEM, Bastion, Vaccine, Security Monitoring 등 다양한 정보보안 서비스를 제공하고 있습니다.

그러나 이러한 서비스가 각각 독립된 항목으로 제공될 경우,  
고객은 각 기능을 스스로 이해하고 조합해야 하며,  
내부 보안팀이나 관제 조직도 이를 실제 위협 흐름과 연결해 신속하게 탐지·분석·대응하기 쉽지 않습니다.

즉, 서비스는 많지만 실제 침해 대응 관점에서는 다음과 같은 구조적 한계가 발생할 수 있습니다.

- 기능은 존재하지만 운영은 분산됨
- 탐지와 대응이 하나의 흐름으로 이어지지 않음
- 고객 스스로 각 서비스를 이해하고 조합해야 함
- 내부 보안팀도 여러 콘솔과 설정을 오가며 판단해야 함
- 결과적으로 “보안 기능 제공”과 “실제 사고 대응력” 사이에 간극이 생김

여기에 비용 문제도 함께 발생합니다.

고객이 웹방화벽, 서버 보안, SIEM, 접근통제, 모니터링, 악성코드 대응 등을 각각 선택할수록  
도입 비용뿐 아니라 운영, 정책 관리, 교육, 분석, 관제 비용까지 함께 증가할 수밖에 없습니다.

즉, 고객 입장에서는 “서비스가 많다”는 것이 곧 “보안이 강하다”는 뜻이 아니라,  
오히려 **선택 부담과 비용 부담이 함께 커지는 구조**가 될 수 있습니다.

반면 **PLURA-XDR**은 다릅니다.

PLURA-XDR은  
**웹 구간 정밀 분석 기술**과  
**EDR 에이전트 설치만으로**  
웹, 서버, 계정, 인증, 위협 행위, 탐지, 대응, 포렌식까지 하나의 체계로 통합 제공합니다.

특히 PLURA-XDR의 차별점은 다음과 같습니다.

- **제로데이 공격 탐지 기술**
- **크리덴셜 스터핑 대응**
- **요청·응답 본문 기반 정밀 분석**
- **민감정보 유출 탐지**
- **웹과 서버를 연결한 상관분석**
- **자동 대응과 포렌식까지 이어지는 운영 구조**

즉, PLURA-XDR은 단순한 웹방화벽이나 EDR이 아니라,

> **보안 기능을 실제 운영 가능한 대응 체계로 바꾸는 통합 정보보안 플랫폼**입니다.

또한 고객 입장에서는,  
NHN Cloud의 여러 보안 서비스를 각각 추가 선택하는 구조보다  
**NHN Cloud 환경에서 실제로 사용 가능한 통합 정보보안 제품 비용**으로 PLURA-XDR을 이해할 수 있습니다.

이는 단순한 가격 비교가 아니라,  
**도입 가능성과 운영 지속 가능성까지 포함한 현실적인 보안 투자 모델**이라는 의미입니다.

## 1️⃣ 문제 인식: 서비스가 많다고 대응이 빨라지는 것은 아닙니다

클라우드 사업자가 제공하는 보안 서비스는 많을수록 좋아 보입니다.  
하지만 실제 운영 현장에서는 다음 질문이 더 중요합니다.

- 이 서비스들이 서로 연결되어 있는가
- 공격 흐름 기준으로 한 번에 파악되는가
- 웹 공격에서 서버 행위까지 이어서 볼 수 있는가
- 탐지 이후 대응까지 즉시 연결되는가
- 고객이 스스로 쉽게 운영할 수 있는가
- 여러 기능을 도입했을 때 비용 부담까지 감당 가능한가

보안 운영은 기능 목록이 아니라 **시간과 연결의 문제**이며, 동시에 **비용과 운영 지속 가능성의 문제**입니다.

예를 들어 웹 공격이 발생했을 때 실제로 필요한 것은 다음과 같습니다.

1. 공격 요청 확인
2. 요청 본문과 응답 결과 분석
3. 서버 행위 연계 확인
4. 반복 시도와 확산 여부 파악
5. 자동 차단 또는 격리
6. 침해 증적 확보

하지만 보안 기능이 각각 따로 존재하면,  
이 흐름은 고객 또는 내부 보안팀이 직접 이어 붙여야 합니다.

그 결과 다음과 같은 문제가 생깁니다.

- 탐지는 되었지만 왜 탐지되었는지 파악이 늦음
- 웹 로그와 서버 로그가 분리되어 사고 흐름이 끊김
- 차단 정책이 수동 운영에 의존함
- 실제 사고 시 여러 콘솔을 오가야 함
- 운영 난이도가 높아짐
- 여러 제품을 동시에 유지해야 하므로 비용 부담도 커짐

즉,

> **분산된 보안 서비스는 제공할 수 있어도,  
> 통합된 대응 경험과 비용 효율을 동시에 제공하지 못할 수 있습니다.**

### Before / After 관점

| 구분 | 다중 개별 서비스 구조 | NHN Cloud + PLURA-XDR |
| --- | --- | --- |
| 운영 화면 | 여러 콘솔 분산 | 통합 흐름 중심 |
| 분석 방식 | 수동 조합 | 상관분석 기반 |
| 사고 대응 | 단계별 수동 전환 | 탐지-대응 연계 |
| 포렌식 확보 | 부분적 | 웹-서버 통합 |
| 도입 판단 | 기능별 선택 부담 큼 | 제품 단위로 이해 쉬움 |
| 비용 설명 | 항목별 누적 | 통합 제품 비용 구조 |

## 2️⃣ NHN Cloud 보안 서비스 구조의 한계

> NHN Cloud는 다양한 정보보안 서비스를 이미 제공하고 있습니다.  
> PLURA-XDR의 장점은 이러한 산재된 기능을 하나의 흐름으로 통합해 보여 주고, 실제 탐지·분석·대응 체계로 연결해 준다는 점에 있습니다.

NHN Cloud의 현재 보안 서비스 구조는 각 영역별 보안 기능을 세분화하여 제공하는 방식입니다.

이 방식은 개별 기능 설명에는 유리할 수 있습니다.  
그러나 고객의 실제 보안 운영 관점에서는 다음과 같은 구조적 부담이 발생합니다.

### 2-1) 고객이 직접 보안 구조를 설계해야 합니다

웹방화벽, 서버 보안, SIEM, DDoS 대응, 접근통제, 모니터링이 따로 존재하면  
고객은 이를 각각 이해하고 조합해 운영해야 합니다.

하지만 대부분의 고객은 다음을 원합니다.

- 기능 목록이 아니라 실제 대응 체계
- 여러 서비스가 아니라 하나의 운영 화면
- 개별 도구가 아니라 연결된 위협 분석
- 탐지 이후 대응까지 이어지는 자동화
- 과도한 선택 부담 없이 도입 가능한 비용 구조

즉 고객은 “보안 서비스 구매”보다  
**운영 가능한 보안 체계 확보**를 원합니다.

### 2-2) 내부 보안팀도 빠른 대응이 어렵습니다

여러 보안 항목이 독립적으로 존재하면  
내부 보안 운영팀이나 SOC 역시 각 서비스를 별도로 확인해야 합니다.

이 경우 실제 침해 대응 시 다음과 같은 비효율이 생깁니다.

- 위협을 하나의 사건으로 묶어 보기 어려움
- 웹 공격과 서버 행위를 연계 분석하기 어려움
- 차단 결정까지 시간이 길어짐
- 탐지와 포렌식이 분리됨
- 운영자 숙련도에 따라 대응 품질 편차가 큼

결국,

> **서비스의 수가 많아질수록  
> 운영 복잡성과 비용 부담도 함께 증가할 수 있습니다.**

### 2-3) 고객은 기능별 과금보다 “실제 쓸 수 있는 보안”을 원합니다

고객 입장에서 여러 서비스를 각각 선택하는 구조는  
기능별 설명에는 명확할 수 있지만, 실제 구매 판단에서는 오히려 부담이 됩니다.

왜냐하면 고객은 다음과 같은 질문을 하게 되기 때문입니다.

- 무엇을 꼭 선택해야 하는가
- 어떤 기능이 중복되는가
- 운영 인력이 충분한가
- 도입 후 실제로 제대로 활용할 수 있는가
- 개별 서비스 비용을 모두 합치면 예산 범위 안에 들어오는가

따라서 고객이 원하는 것은  
“항목별 보안 서비스 가격표”보다  
**실제 운영 가능한 통합 정보보안 제품**입니다.

## 3️⃣ PLURA-XDR의 제안: 보안을 기능이 아니라 운영 체계로 제공합니다

PLURA-XDR은 고객이 여러 보안 기능을 개별적으로 관리하는 구조가 아니라,  
처음부터 **탐지 → 분석 → 대응 → 포렌식이 하나의 흐름으로 연결된 구조**를 제공합니다.

핵심은 단순합니다.

- **웹 구간 위협을 정밀 분석**
- **서버 구간은 EDR 에이전트로 행위 기반 감시**
- **모든 이벤트는 XDR에서 상관분석**
- **탐지 결과는 자동 대응과 포렌식으로 즉시 연결**

즉, 별도의 개별 서비스 나열이 아니라  
**실제 침해 대응을 위한 완성형 운영 모델**을 제공합니다.

또한 이 구조는 고객에게 다음과 같은 장점이 있습니다.

- 여러 보안 서비스를 따로 비교·선택할 필요가 줄어듦
- 제품 간 연동 설계 부담이 낮아짐
- 운영 인력 부담이 줄어듦
- 별도 솔루션 추가 도입 비용을 줄일 수 있음
- 결과적으로 **실제 사용할 수 있는 보안 제품 비용**으로 받아들이기 쉬움

## 4️⃣ NHN Cloud 환경에서의 기술 적용 방식

이번 제안의 핵심은 NHN Cloud의 기존 보안 서비스를 부정하는 것이 아니라,  
기존 기능 위에 **PLURA-XDR을 통합 계층으로 적용하여 가시성과 대응력을 높이는 것**입니다.

### 4-1) 적용 구조

```text
인터넷/외부 공격
   ↓
NHN Cloud 서비스 구간
   ↓
PLURA-Web
 - 요청/응답 본문 분석
 - 제로데이 공격 탐지
 - 크리덴셜 스터핑 탐지
   ↓
PLURA-XDR
 - 웹 이벤트 상관분석
 - 인증/세션 이상행위 분석
 - 기존 보안 이벤트 연계
   ↓
PLURA-EDR
 - 프로세스/파일/네트워크 행위 탐지
 - 웹셸/확산 행위 탐지
   ↓
자동 대응 / 알림 / 포렌식
```

### 4-2) 기존 NHN Cloud 보안 서비스와의 관계

PLURA-XDR은 다음 두 가지 방식으로 적용할 수 있습니다.

| 적용 방식                  | 설명                                                                                         |
| ---------------------- | ------------------------------------------------------------------------------------------ |
| Co-exist 모델            | 기존 NHN Cloud Web Firewall, SIEM, Security Monitoring을 유지하면서 PLURA-XDR이 정밀 분석 및 상관분석 계층을 추가 |
| Replace/Consolidate 모델 | 고객 환경에 따라 일부 개별 기능을 PLURA-XDR로 통합해 운영 화면과 대응 체계를 단순화                                       |

즉, NHN Cloud 서비스와 PLURA-XDR은 반드시 대체 관계만이 아니라  
**단계적 공존과 통합 전환이 가능한 구조**로 제안할 수 있습니다.

### 4-3) 로그 수집 및 연계 방식

PLURA-XDR은 다음 데이터를 중심으로 통합 분석합니다.

| 데이터 유형 | 수집/분석 대상                          | 방식       |
| ------ | --------------------------------- | -------- |
| 웹 구간   | 요청 본문, 응답 본문, URL, 세션, 인증 이벤트     | 웹 구간 분석  |
| 서버 구간  | 프로세스, 파일, 네트워크, 계정 행위             | EDR 에이전트 |
| 운영 이벤트 | 정책 위반, 차단 이벤트, 알림, 포렌식 증적         | XDR 연계   |
| 연계 이벤트 | 기존 SIEM/Security Monitoring 상위 경보 | 이벤트 연동   |

이를 통해 NHN Cloud의 Security Monitoring 또는 기존 SIEM이 제공하는 상위 가시성에 더해,  
PLURA-XDR은 **실제 공격의 세부 흐름**을 분석하는 계층 역할을 수행할 수 있습니다.

### 4-4) 콘솔과 운영 연계

운영 방식은 다음과 같이 제안할 수 있습니다.

* 1차 운영: PLURA-XDR 통합 콘솔에서 웹-서버 상관분석 수행
* 2차 연계: NHN Cloud 운영팀 또는 고객 SOC에 알림 전달
* 3차 대응: 정책 기반 자동 차단 또는 운영 승인 후 조치

즉, NHN Cloud 콘솔을 무조건 대체하는 것이 아니라  
**실제 대응을 위한 전문 분석 콘솔**로 PLURA-XDR을 위치시킬 수 있습니다.

### 4-5) 서버 및 Kubernetes 배포 방식

PLURA-EDR은 NHN Cloud의 다음 환경에서 단계적으로 적용 가능합니다.

| 환경                     | 적용 방식          |
| ---------------------- | -------------- |
| VM 인스턴스                | 에이전트 직접 설치     |
| Baremetal 서버           | 에이전트 직접 설치     |
| Kubernetes Worker Node | 노드 단위 배포 및 운영  |
| Windows/Linux 혼합       | OS별 에이전트 정책 적용 |
| Hybrid 구성 서버           | 단계별 확대 적용      |

초기 단계에서는 인터넷 노출 웹 서버 또는 핵심 API 서버 중심으로 적용하고,  
이후 필요 시 전체 서버군으로 확대하는 방식이 현실적입니다.

## 5️⃣ 왜 PLURA-XDR이 더 실질적인가

### 5-1) 설치와 운영이 단순합니다

PLURA-XDR은 다음 두 가지로 시작할 수 있습니다.

1. **웹 구간 보안 분석 적용**
2. **서버에 EDR 에이전트 설치**

이 두 가지 만으로 다음 기능이 즉시 통합 제공됩니다.

* 웹 공격 탐지
* 요청/응답 기반 정밀 분석
* 인증/세션 이상행위 탐지
* 서버 행위 기반 위협 탐지
* 웹셸 및 침해 징후 탐지
* 사고 연계 분석
* 자동 대응
* 포렌식 로그 확보

### 5-2) 탐지와 대응이 분리되지 않습니다

PLURA-XDR은 탐지 자체보다 그 이후를 더 중요하게 봅니다.

* 공격 요청을 탐지
* 관련 서버 행위를 즉시 상관분석
* 동일 위협의 반복성 확인
* 정책 기반 자동 차단
* 사고 흔적과 증적을 동시에 확보

즉,

> **탐지, 대응, 포렌식이 따로 노는 구조가 아니라 하나로 이어집니다.**

### 5-3) 고객 스스로 운영하기 쉽습니다

운영 화면과 분석 방식이 통합되어 있어,

* 고객 스스로 운영 가능
* 내부 보안팀의 업무 효율 향상
* SOC 대응 속도 향상
* 교육과 인수인계 부담 감소
* 여러 솔루션 간 역할 중복 감소

이는 단순한 편의성의 문제가 아니라  
**총 운영비용과 실제 사고 대응력의 문제**입니다.

### 5-4) 비용 관점에서도 현실적입니다

고객이 개별 보안 서비스를 하나씩 추가 선택하는 구조에서는  
기능이 늘어날수록 예산 부담도 커질 수밖에 없습니다.

반면 PLURA-XDR은  
웹 보안, 서버 행위 분석, 통합 탐지, 자동 대응, 포렌식까지 하나의 체계로 제공하기 때문에,  
고객은 이를 다수 서비스의 누적 비용이 아니라 **통합 정보보안 제품 비용**으로 이해할 수 있습니다.

## 6️⃣ 왜 기존 웹방화벽과 다른가: 제로데이 공격 탐지와 응답 본문 기반 분석의 차이

클라우드 환경에서 웹방화벽은 기본적으로 리버스 프록시 방식으로 동작합니다.  
따라서 여기서 중요한 차이는 “리버스 프록시인가 아닌가”가 아닙니다.

진짜 차이는 다음입니다.

* 알려진 패턴 기반 탐지에 머무르는가
* 아니면 **제로데이 공격까지 탐지할 수 있는가**
* 요청만 보는가
* 아니면 **응답 본문까지 분석하여 민감정보 유출 징후를 식별할 수 있는가**
* 단순 URL/IP/응답코드 중심인가
* 아니면 **인증·세션·행위 흐름까지 연계 분석할 수 있는가**

### 6-1) 제로데이 공격 탐지는 기존 웹방화벽과 본질적으로 다른 영역입니다

대부분의 웹방화벽은 알려진 공격 패턴, 시그니처, 정해진 룰셋 중심으로 동작합니다.  
이 방식은 이미 알려진 공격을 방어하는 데에는 유효하지만,  
새로운 우회 기법이나 변형된 공격, 즉 **제로데이 공격**에는 한계가 있습니다.

반면 PLURA-XDR은

* 웹 요청과 응답의 흐름
* 정상 행위 대비 편차
* 반복성과 자동화 패턴
* 인증 및 세션 이상 징후
* 서버 행위와의 상관관계

를 함께 분석함으로써,  
단순 룰 매칭이 아니라 **행위 기반의 제로데이 탐지**까지 확장할 수 있습니다.

### 6-2) 크리덴셜 스터핑 대응은 웹 보안의 핵심 경쟁력입니다

오늘날 대규모 웹 침해 사고의 상당수는  
취약점 공격 그 자체보다 **탈취 계정 기반 공격**, 즉 크리덴셜 스터핑에서 시작됩니다.

이 공격은 다음과 같은 특성이 있습니다.

* 정상 로그인 요청처럼 보일 수 있음
* 다양한 IP로 분산될 수 있음
* URL과 응답코드만으로는 식별이 어려움
* 인증 실패와 성공 흐름, 세션 변화, 반복 시도 패턴을 함께 봐야 함

따라서 크리덴셜 스터핑 대응은  
단순한 차단 룰이 아니라 다음 분석이 가능해야 합니다.

* 계정 단위 반복 시도 탐지
* 세션/토큰 발급 흐름 분석
* 성공·실패 결과의 이상 징후 식별
* 자동화 패턴 탐지
* 정책 기반 자동 차단 연계

### 6-3) 민감정보탐지는 “응답 본문 분석” 여부가 핵심입니다

민감정보탐지를 제공한다고 하더라도,  
그 기능이 실제로 **응답 본문 분석**을 포함하지 않는다면 실질적인 데이터 유출 탐지에는 한계가 있습니다.

> 만약 제공되는 민감정보탐지 기능이 응답 본문 분석까지 포함한다면,  
> 이는 당사가 보유한 응답 본문 기반 보안 분석 특허와의 관계를 별도로 검토할 필요가 있습니다.

### 6-4) 경쟁 솔루션 대비 차별점 요약

| 구분           | 일반 웹방화벽     | 일반 XDR/EDR | NHN 기존 서비스 조합 | PLURA-XDR |
| ------------ | ----------- | ---------- | ------------- | --------- |
| 알려진 공격 탐지    | 가능          | 일부 가능      | 가능            | 가능        |
| 제로데이 공격 탐지   | 제한적         | 서버 행위 중심   | 제한적           | 웹+서버 연계   |
| 요청 본문 분석     | 제한적         | 없음         | 제한적           | 가능        |
| 응답 본문 분석     | 드묾          | 없음         | 제한적           | 가능        |
| 크리덴셜 스터핑 대응  | 제한적         | 제한적        | 분산 대응         | 세션 흐름 기반  |
| 웹-서버 상관분석    | 제한적         | 제한적        | 운영자 수동 조합     | 가능        |
| 자동 대응+포렌식 연계 | 부분적         | 부분적        | 부분적           | 통합 제공     |
| 운영 화면        | 개별 장비/기능 중심 | 단말 중심      | 다중 콘솔         | 통합 흐름 중심  |

즉,

> 기존 웹방화벽이 “알려진 공격을 막는 도구”에 가깝다면,  
> PLURA-XDR은 **알려지지 않은 공격, 계정 기반 공격, 데이터 유출까지 탐지하고 대응하는 통합 플랫폼**입니다.

3. 관련 서버 2~5대 수준 EDR 적용
4. 크리덴셜 스터핑 및 웹-서버 연계 시나리오 검증

즉, 처음부터 전체 전환이 아니라  
**작게 시작해 빠르게 검증하고 단계적으로 확대하는 방식**이 적합합니다.

## 9️⃣ 가격 모델과 비용 논리

PLURA-XDR의 가격 모델은 고객이 이해하기 쉬운 형태로 제안하는 것이 중요합니다.

### 9-1) 기본 가격 구조 예시

* 웹 서비스 수 또는 처리 구간 기준
* 서버 대수 기준 EDR 과금
* 통합 XDR 분석 기능 포함
* 초기 구축/온보딩 비용 별도 또는 프로모션 포함

### 9-2) 고객 관점의 비용 메시지

고객은 여러 보안 서비스를 각각 선택할수록  
기능별 과금, 운영 인력, 분석 시간, 교육 비용까지 함께 부담하게 됩니다.

반면 PLURA-XDR은

* 웹 구간 분석
* 서버 행위 분석
* 통합 XDR 상관분석
* 자동 대응
* 포렌식

을 하나의 체계로 제공하므로,  
고객은 이를 **복수 서비스의 합산 비용**이 아니라  
**바로 사용할 수 있는 통합 정보보안 제품 비용**으로 이해할 수 있습니다.

### 9-3) 제안 시 사용할 수 있는 비용 표현

* “기존 개별 서비스 N개를 별도로 도입·운영하는 구조보다 단순한 비용 구조”
* “운영 인력과 분석 시간을 포함한 총비용 관점에서 유리”
* “기능별 선택형 보안이 아니라 실제 활용 가능한 통합 제품 비용”
* “초기 도입 장벽을 낮추고 이후 단계적 확장이 가능한 모델”

### 9-4) 비용 절감 표현 예시

제안 단계에서는 아래처럼 표현할 수 있습니다.

> 고객 환경에 따라 차이가 있으나, 개별 보안 서비스 다중 선택 구조 대비  
> **도입 항목 단순화와 운영 인력 절감을 통해 총비용 기준 20~30% 수준의 효율 개선 가능성**을 기대할 수 있습니다.

또는 보다 적극적인 제안 문구로는 다음과 같이 사용할 수 있습니다.

> 기존 웹방화벽 + 서버보안 + SIEM 조합 대비,  
> 고객 환경에 따라 **30% 내외의 비용 효율화 검토가 가능한 통합 구조**로 제안할 수 있습니다.

※ 위 수치는 제안용 예시 범위이며, 실제 절감률은 대상 범위와 기존 운영 구조에 따라 별도 산정합니다.

## 한 문장 정리

> NHN Cloud의 개별 보안 서비스가 “기능의 집합”이라면,  
> PLURA-XDR은 이를 **제로데이 탐지, 크리덴셜 스터핑 대응, 민감정보 유출 탐지까지 포함하는 실제 대응 가능한 통합 운영 체계이자, 고객이 현실적으로 도입 가능한 정보보안 제품**으로 완성하는 플랫폼입니다.

---

## 결론

지금 고객이 원하는 것은 보안 기능의 나열이 아닙니다.  
고객이 원하는 것은 적은 인력으로도 빠르게 운영할 수 있고,  
실제 해킹 상황에서 즉시 탐지하고 대응할 수 있으며,  
비용적으로도 감당 가능한 통합 보안입니다.

PLURA-XDR은  
웹 구간에 대한 정밀 분석과  
EDR 에이전트 설치만으로  
웹, 서버, 인증, 공격 탐지, 자동 대응, 포렌식까지 하나의 체계로 제공합니다.

특히 그 차별점은

* **제로데이 공격 탐지**
* **크리덴셜 스터핑 대응**
* **응답 본문 기반 민감정보 유출 탐지**
* **웹과 서버를 연결한 상관분석**
* **탐지 이후 자동 대응과 포렌식 연계**
* **고객이 실제 도입 가능한 비용 구조**

에 있습니다.

따라서 PLURA-XDR은

* 고객에게는 더 쉬운 운영을,
* 보안팀에게는 더 빠른 대응을,
* NHN Cloud에는 더 강한 경쟁력을

제공할 수 있는 가장 현실적인 통합 보안 서비스입니다.

## 부록 B: 제안 핵심 메시지

* NHN Cloud는 다양한 보안 기능을 제공하고 있습니다.
* 그러나 개별 서비스 분산 구조만으로는 실제 침해 대응 흐름을 완성하기 어렵습니다.
* 고객이 여러 서비스를 각각 선택할수록 비용 부담과 운영 부담도 함께 증가할 수 있습니다.
* PLURA-XDR은 웹과 서버를 통합 분석하고 자동 대응까지 연결합니다.
* 특히 제로데이 공격 탐지, 크리덴셜 스터핑 대응, 응답 본문 기반 민감정보 유출 탐지에서 차별성을 가집니다.
* 고객 스스로 또는 내부 보안 관제팀이 더 효율적으로 운영할 수 있습니다.
* 고객 입장에서는 여러 선택형 보안 서비스가 아니라, 실제로 사용 가능한 통합 정보보안 제품 비용으로 이해할 수 있습니다.
* 정량 KPI와 단계별 로드맵을 통해 PoC 및 실제 도입 가능성을 검증할 수 있습니다.
* 이는 어떤 클라우드 서비스보다 경쟁력 있는 통합 보안 모델이 될 수 있습니다.
