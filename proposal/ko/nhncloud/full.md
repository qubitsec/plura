# PLURA-XDR 제안서
## NHN Cloud 보안 서비스 통합 경쟁력 강화 제안

## 0️⃣ 제안 요약

NHN Cloud는 App Security Check, Server Security Check, Web Firewall, DDoS Guard, SIEM, Bastion, Vaccine, Security Monitoring 등 다양한 정보보안 서비스를 제공하고 있습니다.

그러나 이러한 서비스가 개별 항목으로 분리되어 제공될 경우, 고객은 각 기능을 스스로 이해하고 연계하여 운영해야 하며, 내부 보안팀이나 관제 조직 역시 위협 흐름과 연결하여 신속하게 탐지·분석·대응하기가 쉽지 않습니다.

즉, 서비스는 많지만 실제 침해 대응 관점에서는 다음과 같은 구조적 한계가 발생할 수 있습니다.

- 기능은 존재하지만 운영이 분산됨
- 탐지와 대응이 하나의 흐름으로 이어지지 않음
- 고객이 각 서비스를 직접 조합해야 함
- 내부 보안팀도 여러 콘솔과 정책을 오가며 판단해야 함
- 결과적으로 보안 기능의 제공과 실제 사고 대응력 사이에 간극이 생김

반면 PLURA-XDR은 다릅니다.

PLURA-XDR은  
""**웹 구간의 위협 탐지**"",  
""**크리덴셜 스터핑 공격 대응**"",  
""**제로데이 공격 탐지**"",  
""**서버 행위 기반 위협 분석**"",  
""**자동 대응**"",  
""**포렌식 로그 확보**""를  
하나의 체계로 통합 제공합니다.

고객은 더 이상 여러 보안 서비스를 각각 연결해 이해할 필요가 없습니다.  
고객 스스로도, 내부 보안 관제팀도, 더 적은 인력으로 더 빠르게 운영할 수 있습니다.

즉, PLURA-XDR은 단순히 개별 보안 기능을 더하는 제품이 아니라,

> ""**보안 기능을 실제 운영 가능한 대응 체계로 바꾸는 통합 정보보안 플랫폼**""입니다.

## 1️⃣ 문제 인식: 서비스가 많다고 대응이 빨라지는 것은 아닙니다

클라우드 사업자가 제공하는 보안 서비스는 많을수록 좋아 보입니다.  
하지만 실제 운영 현장에서는 다음 질문이 더 중요합니다.

- 이 서비스들이 서로 연결되어 있는가
- 공격 흐름 기준으로 한 번에 파악되는가
- 웹 공격에서 서버 행위까지 이어서 볼 수 있는가
- 탐지 이후 대응까지 즉시 연결되는가
- 고객이 스스로 쉽게 운영할 수 있는가

보안 운영은 기능 목록이 아니라 ""**시간과 연결의 문제**""입니다.

예를 들어 웹 공격이 발생했을 때 실제로 필요한 것은 다음과 같습니다.

1. 공격 요청이 무엇이었는지 확인
2. 요청 본문과 응답 결과를 분석
3. 해당 요청이 서버에서 어떤 행위로 이어졌는지 확인
4. 동일 공격자의 반복 시도 여부 확인
5. 자동 차단 또는 격리 수행
6. 침해 증적을 즉시 확보

하지만 보안 기능이 각각 따로 존재하면, 이 흐름은 고객 또는 내부 보안팀이 직접 이어 붙여야 합니다.

그 결과 다음과 같은 문제가 생깁니다.

- 탐지는 되었지만 왜 탐지되었는지 파악이 늦음
- 웹 로그와 서버 로그가 분리되어 사고 흐름이 끊김
- 차단 정책이 수동 운영에 의존함
- 실제 사고 시 여러 서비스 콘솔을 오가야 함
- 고객 입장에서 운영 난이도가 높아짐

즉,

> ""**분산된 보안 서비스는 제공할 수 있어도, 통합된 대응 경험은 제공하지 못할 수 있습니다.**""

## 2️⃣ NHN Cloud 보안 서비스 구조의 한계

NHN Cloud의 현재 보안 서비스 구조는 각 영역별 보안 기능을 세분화하여 제공하는 방식입니다.

이 방식은 개별 기능 설명에는 유리할 수 있습니다.  
그러나 고객의 실제 보안 운영 관점에서는 다음과 같은 구조적 부담이 발생합니다.

### 2-1) 고객이 직접 보안 구조를 설계해야 합니다

웹방화벽, 서버 보안, SIEM, DDoS 대응, 접근통제, 모니터링이 따로 존재하면 고객은 이를 각각 이해하고 조합해 운영해야 합니다.

하지만 대부분의 고객은 다음을 원합니다.

- 기능 목록이 아니라 실제 대응 체계
- 여러 서비스가 아니라 하나의 운영 화면
- 개별 도구가 아니라 연결된 위협 분석
- 탐지 이후 대응까지 이어지는 자동화

즉 고객은 보안 서비스 구매보다  
""**운영 가능한 보안 체계 확보**""를 원합니다.

### 2-2) 내부 보안팀도 빠른 대응이 어렵습니다

여러 보안 항목이 독립적으로 존재하면 내부 보안 운영팀이나 SOC 역시 각 서비스를 별도로 확인해야 합니다.

이 경우 실제 침해 대응 시 다음과 같은 비효율이 생깁니다.

- 위협을 하나의 사건으로 묶어 보기 어려움
- 웹 공격과 서버 행위를 연계 분석하기 어려움
- 차단 결정까지 시간이 길어짐
- 탐지와 포렌식이 분리됨
- 운영자 숙련도에 따라 대응 품질 편차가 큼

결국,

> ""**서비스의 수가 많아질수록 운영 복잡성도 함께 증가할 수 있습니다.**""

## 3️⃣ PLURA-XDR의 제안: 보안을 기능이 아니라 운영 체계로 제공합니다

PLURA-XDR은 고객이 여러 보안 기능을 개별적으로 관리하는 구조가 아니라, 처음부터 ""**탐지-분석-대응-포렌식이 하나의 흐름으로 연결된 구조**""를 제공합니다.

핵심은 단순합니다.

- 웹 구간의 요청과 응답을 기반으로 위협을 분석
- 인증·세션 행위를 기반으로 크리덴셜 스터핑 공격을 탐지
- 알려진 패턴이 아닌 이상 행위를 기반으로 제로데이 공격을 식별
- 서버 구간은 EDR 에이전트로 행위 기반 감시
- 모든 이벤트는 XDR에서 상관분석
- 탐지 결과는 자동 대응과 포렌식으로 즉시 연결

즉, 별도의 개별 서비스 나열이 아니라  
""**실제 침해 대응을 위한 완성형 운영 모델**""을 제공합니다.

## 4️⃣ 왜 PLURA-XDR이 더 실질적인가

### 4-1) 설치와 운영이 단순합니다

PLURA-XDR은 다음 두 가지로 시작할 수 있습니다.

1. 웹 서비스 앞단 적용
2. 서버에 EDR 에이전트 설치

이 두 가지 만으로 다음 기능이 즉시 통합 제공됩니다.

- 웹 공격 탐지
- 요청/응답 기반 정밀 분석
- 인증/세션 이상행위 탐지
- 크리덴셜 스터핑 공격 대응
- 제로데이 공격 탐지
- 서버 행위 기반 위협 탐지
- 웹셸 및 침해 징후 탐지
- 사고 연계 분석
- 자동 대응
- 포렌식 로그 확보

즉, 고객은 별도의 다수 보안 서비스를 조합하지 않아도  
""**실질적인 통합 보안 운영**""을 바로 시작할 수 있습니다.

### 4-2) 탐지와 대응이 분리되지 않습니다

보안 운영에서 가장 큰 문제 중 하나는  
""**탐지는 되었지만 대응은 늦는**"" 구조입니다.

PLURA-XDR은 탐지 자체보다 그 이후를 더 중요하게 봅니다.

- 공격 요청을 탐지
- 응답 결과를 함께 확인
- 관련 서버 행위를 즉시 상관분석
- 동일 위협의 반복성과 자동화 여부를 확인
- 정책 기반 자동 차단
- 사고 흔적과 증적을 동시에 확보

즉,

> ""**탐지, 대응, 포렌식이 따로 노는 구조가 아니라 하나로 이어집니다.**""

### 4-3) 고객 스스로 운영하기 쉽습니다

보안 서비스가 많을수록 고객은 보안 전문가에 더 의존하게 됩니다.  
하지만 PLURA-XDR은 오히려 반대입니다.

운영 화면과 분석 방식이 통합되어 있어,

- 고객 스스로 운영 가능
- 내부 보안팀의 업무 효율 향상
- SOC 대응 속도 향상
- 교육과 인수인계 부담 감소
- 여러 솔루션 간 역할 중복 감소

이는 단순한 편의성의 문제가 아니라  
""**총 운영비용과 실제 사고 대응력의 문제**""입니다.

## 5️⃣ 웹 보안의 핵심은 구조가 아니라 ""**제로데이 탐지 역량**""입니다

NHN Cloud 환경에서 웹 서비스는 이미 클라우드 구조상 프록시 기반으로 운영됩니다.  
따라서 여기서 중요한 것은 구조 자체가 아닙니다.

실제 경쟁력의 차이는  
""**그 구조 위에서 무엇을 탐지할 수 있는가**""에 있습니다.

많은 기존 웹방화벽은 알려진 공격 패턴, 시그니처, 정형화된 룰 기반 차단에는 대응할 수 있습니다.  
그러나 실제 침해 사고에서는 알려진 공격보다  
""**정상 요청처럼 보이지만 실제로는 공격인 행위**""  
또는  
""**기존 룰로 정의되지 않은 제로데이 공격**""  
을 얼마나 빠르게 식별할 수 있는지가 더 중요합니다.

PLURA-XDR의 핵심은 단순 웹방화벽 기능이 아니라,  
""**웹 요청과 응답 전체 흐름을 기반으로 제로데이 공격을 탐지할 수 있는 분석 기술**""이 포함되어 있다는 점입니다.

예를 들어 다음과 같은 공격은 단순 URL, IP, 응답코드만으로는 충분히 식별하기 어렵습니다.

- 알려지지 않은 입력값 변조 공격
- 정상 API 호출처럼 위장한 데이터 유출 시도
- 응답 결과를 기반으로 반복 최적화되는 자동화 공격
- 인증 우회 및 세션 악용 공격
- 웹셸 업로드 이후의 연계 행위
- JWT 또는 세션 탈취 후의 비정상 접근

이러한 공격은 Access Log 중심 분석만으로는 한계가 있으며,  
""**요청 본문과 응답 결과를 함께 분석할 수 있어야**""  
비정상 행위를 실질적으로 식별할 수 있습니다.

특히 응답 본문 분석은 단순 부가 기능이 아니라,  
""**공격 성공 여부, 데이터 노출 여부, 비정상 응답 패턴을 판단하는 핵심 근거**""가 될 수 있습니다.

따라서 NHN Cloud의 보안 경쟁력도 단순히 다양한 보안 서비스를 나열하는 수준을 넘어,  
""**제로데이 공격까지 탐지 가능한 정밀 분석 체계를 제공하는가**""로 평가되어야 합니다.

PLURA-XDR은 바로 이 지점에서 차별화됩니다.

- 요청 본문 기반 분석
- 응답 결과 기반 분석
- 웹과 서버 행위의 상관분석
- 알려진 공격뿐 아니라 제로데이 행위 탐지
- 탐지 이후 자동 대응과 포렌식 연계

또한 만약 NHN Cloud가 제공하는 ""**민감정보탐지**"" 기능이 단순 패턴 검색 수준을 넘어  
""**응답 본문 전체를 분석하여 민감정보 노출 여부를 식별하는 방식**""까지 포함하고 있다면,  
이는 당사의 특허 기술 범위와의 관계에 대해 별도 검토가 필요한 사안이며, 경우에 따라서는 ""**특허 위반 소지**""가 있을 수 있다는 점도 신중히 살펴볼 필요가 있습니다.

결국 중요한 것은 구조가 아니라,

> ""**어디까지 볼 수 있는가**"",  
> ""**무엇을 탐지할 수 있는가**"",  
> 그리고 ""**제로데이 공격까지 실제로 대응할 수 있는가**""입니다.

## 6️⃣ ""**크리덴셜 스터핑 공격 대응**""은 기존 웹방화벽과 PLURA-XDR의 핵심 차이점입니다

기존 웹방화벽은 대체로 알려진 공격 패턴을 기반으로 요청을 차단하는 데 강점이 있습니다.  
하지만 크리덴셜 스터핑 공격은 단순한 웹 취약점 공격과 다릅니다.

이 공격은 정상 로그인 또는 정상 API 요청처럼 보이면서,  
탈취된 계정 정보나 인증 정보를 대량으로 자동화 시도하는 방식으로 진행됩니다.

즉, 문제는 단순 요청 자체가 아니라  
""**반복성**"",  
""**자동화 패턴**"",  
""**성공과 실패의 비율**"",  
""**세션 및 인증 흐름의 이상 징후**""에 있습니다.

따라서 크리덴셜 스터핑 공격 대응에는 다음이 필요합니다.

- 요청 본문 또는 인증 파라미터 분석
- 로그인·인증·토큰 발급 흐름 분석
- 성공/실패 결과 신호 분석
- 동일 계정, 세션, 엔드포인트 기준 반복 시도 탐지
- 자동화 행위의 시간 밀도와 편차 분석
- 정책 기반 제한, 추가 인증, 차단

이 점이 기존 웹방화벽과 PLURA-XDR의 중요한 차이입니다.

기존 웹방화벽이  
""**룰 기반 요청 차단**""에 무게를 둔다면,

PLURA-XDR은  
""**인증 행위 기반의 크리덴셜 스터핑 탐지와 대응**""까지 포함합니다.

즉, PLURA-XDR은 단순히 공격 문자열을 찾는 것이 아니라,  
""**정상처럼 보이는 인증 시도 속에서 공격 패턴을 식별하고 차단하는 체계**""를 제공합니다.

이는 대규모 로그인 서비스, API 기반 서비스, 고객 계정 중심 서비스에서 특히 중요한 차별점입니다.

## 7️⃣ PLURA-XDR이 NHN Cloud 경쟁력을 높이는 이유

PLURA-XDR은 단순히 하나의 추가 보안 서비스가 아닙니다.  
NHN Cloud의 전체 보안 경쟁력을 재정의할 수 있는 구조입니다.

### 7-1) ""**서비스 수 경쟁**""이 아니라 ""**운영 완성도 경쟁**""입니다

많은 클라우드 사업자가 보안 서비스를 다양하게 나열합니다.  
그러나 고객이 실제로 체감하는 경쟁력은 다음입니다.

- 얼마나 쉽게 운영되는가
- 얼마나 빨리 탐지되는가
- 얼마나 빠르게 대응되는가
- 실제 사고 시 증적 확보가 가능한가
- 보안팀과 SOC가 효율적으로 일할 수 있는가

PLURA-XDR은 이 질문에 직접 답할 수 있는 플랫폼입니다.

### 7-2) 고객에게 더 강한 메시지를 줄 수 있습니다

NHN Cloud가 PLURA-XDR과 같은 통합 보안 체계를 제공할 경우, 고객에게 다음과 같이 설명할 수 있습니다.

- 개별 보안 서비스를 따로 운영할 필요가 없습니다
- 웹과 서버를 통합한 XDR 체계를 제공합니다
- 제로데이 공격 탐지 역량을 제공합니다
- 크리덴셜 스터핑 공격 대응까지 지원합니다
- 탐지 이후 대응과 포렌식까지 하나로 제공합니다
- 고객 내부 보안팀의 운영 부담을 줄입니다
- 실제 침해 대응 시간 단축에 유리합니다

즉, 단순 기능 소개가 아니라  
""**우리는 실제 대응 가능한 클라우드 보안을 제공한다**""는 메시지를 만들 수 있습니다.

## 1️⃣1️⃣ 결론

지금 고객이 원하는 것은 보안 기능의 나열이 아닙니다.  
고객이 원하는 것은 적은 인력으로도 빠르게 운영할 수 있고, 실제 해킹 상황에서 즉시 탐지하고 대응할 수 있는 통합 보안입니다.

PLURA-XDR은 웹, 서버, 인증, 공격 탐지, 자동 대응, 포렌식까지 하나의 체계로 제공합니다.

특히 PLURA-XDR의 차별점은 다음 두 가지입니다.

- ""**제로데이 공격 탐지 역량**""
- ""**크리덴셜 스터핑 공격 대응 역량**""

이는 기존 웹방화벽 업체와 구분되는 핵심 차이점이며, NHN Cloud가 고객에게 더 강한 보안 경쟁력을 제안할 수 있게 해 주는 현실적인 기반입니다.

따라서 PLURA-XDR은

- 고객에게는 더 쉬운 운영을,
- 보안팀에게는 더 빠른 대응을,
- NHN Cloud에는 더 강한 경쟁력을

제공할 수 있는 가장 현실적인 통합 보안 서비스입니다.

- 고객 스스로 또는 내부 보안 관제팀이 더 효율적으로 운영할 수 있습니다.
- 이는 어떤 클라우드 서비스보다 경쟁력 있는 통합 보안 모델이 될 수 있습니다.
