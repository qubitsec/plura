# PLURA-XDR 기술 비교 제안서
## SentinelOne 대비 PLURA-XDR의 비교 관점

> 본 문서는 XDR 전체 항목에 대한 전면 비교 문서가 아닙니다.  
> 비교 범위는 아래 기술한 기준에 따라 한정하였습니다.

---

## 핵심 요약

SentinelOne과 PLURA-XDR은 모두 XDR 범주에 포함될 수 있지만,  
실제 분석의 출발점과 강점 영역은 다릅니다.

SentinelOne은 엔드포인트 중심의 탐지와 자동화 구조를 기반으로 하며,  
PLURA-XDR은 웹 요청·응답 구간, 인증/세션 흐름, 서버 행위, 자동 대응, 포렌식을 연결하는 구조를 기반으로 합니다.  
따라서 두 제품은 같은 영역을 모두 동일한 방식으로 다룬다기보다, **강하게 설명할 수 있는 구간이 다르다**고 보는 것이 더 정확합니다.

일반적인 기업 환경, 특히 웹 서비스나 API 서비스 비중이 높은 조직에서는 다음 비교 축이 중요합니다.

- 크리덴셜 스터핑과 같은 대규모 인증 공격을 어떤 근거로 식별하는가
- 인증키 유출 이후 정상처럼 보이는 공격을 어떻게 탐지하는가
- 제로데이 또는 비정형 웹 공격을 어느 구간에서 포착하는가
- 웹 요청과 응답, 인증 결과, 서버 후속 행위, 자동 대응, 포렌식까지 하나의 사건 흐름으로 연결할 수 있는가
- 웹셸 업로드 이후 LOLBAS 또는 GTFOBins 기반 공격 전환을 어디까지 설명할 수 있는가

PLURA-XDR은 이 기준에서  
**웹 요청 본문 + 웹 응답 본문/결과 신호 + 서버 감사 로그 및 행위 로그**를 결합해 분석한다는 점에서 차별성이 있습니다.  
또한 관련 제안 문서들에서도 PLURA-XDR의 핵심은 요청·응답 본문 기반 분석, 웹-서버 상관분석, 자동 대응과 포렌식 연계로 반복해 설명됩니다.

다만 이 문서는 전체 XDR 시장에서의 절대적 우열을 단정하려는 문서가 아닙니다.  
비교의 초점은 **일반적인 기업 환경에서 어떤 제품이 어떤 공격 표면을 더 직접적으로 설명할 수 있는가**에 있습니다.

---

## 비교 기준

비교 기준은 다음과 같습니다.

### 1. 탐지의 출발점
공격을 어떤 데이터에서 먼저 포착하는가

### 2. 탐지 근거의 깊이
왜 공격이라고 판단했는지 어느 수준까지 설명 가능한가

### 3. 공격 흐름 복원 능력
웹 요청, 인증 결과, 서버 후속 행위, 자동 대응, 포렌식을 하나의 사건으로 연결할 수 있는가

### 4. 운영 적용성
기업 서비스 환경에서 보안팀이 실질적으로 운영 가능한 구조인가

### 5. 대응 연계성
탐지 이후 제한, 차단, 격리, 포렌식으로 자연스럽게 이어질 수 있는가

## 구조적 차이

### SentinelOne의 중심 축

SentinelOne은 엔드포인트 중심 EDR/XDR 구조로서 다음 영역이 중심입니다.

- 행동 기반 엔드포인트 탐지
- Storyline 기반 사건 흐름 정리
- 자동 대응 및 롤백
- 생성형 AI 기반 조사 보조
- AI SIEM 및 데이터 파이프라인 확장

기존 분석 문서에서도 SentinelOne은 단일한 “마법 같은 AI”라기보다,  
**행동 기반 탐지 + 자동화 + 생성형 분석 보조가 결합된 다층 구조**로 설명됩니다.  
따라서 전통적인 시그니처 기반 제품으로 단순화해서 보는 것은 적절하지 않습니다.

### PLURA-XDR의 중심 축

PLURA-XDR은 다른 출발점을 가집니다.  
엔드포인트 결과를 중심으로 보기보다,  
**웹 요청 본문과 응답 결과 신호를 먼저 보고, 이를 인증/세션 흐름과 서버 로그, 자동 대응, 포렌식으로 연결**합니다.

이 차이는 다음 영역에서 의미를 가집니다.

- 크리덴셜 스터핑
- 인증키 기반 Brute Force
- 제로데이
- 웹셸 업로드 이후 LOLBAS / GTFOBins 전환
- 데이터 유출 징후 해석

특히 대규모 웹/API 서비스 환경에서는 Brute Force와 크리덴셜 스터핑의 근거가 Access Log 표면이 아니라 대부분 **요청 본문과 응답 결과 신호** 안에 있으며, 키를 가진 공격자 시나리오까지 포함한 2차 방어선으로 행위 기반 탐지·차단이 필요합니다.  
이러한 관점은 쿠팡과 같은 대규모 서비스 환경에서도 그대로 적용될 수 있습니다.

## 크리덴셜 스터핑과 인증키 기반 Brute Force

일반적인 기업 환경에서 Brute Force는 단순 로그인 실패 누적과 동일하지 않습니다.  
특히 인증키, 서명키, API Key, 토큰이 외부로 노출된 이후에는 헤더나 서명 수준에서는 정상 요청처럼 보일 수 있습니다.

이때 중요한 핵심은 다음과 같습니다.

- 키 교체/폐기는 필수지만, 교체 전후 공백과 지연, 자동화 공격까지 모두 막지는 못함
- 따라서 “키를 가진 공격자” 시나리오까지 포함하는 2차 방어선이 필요함
- 그 방어선의 핵심은 **요청 본문 + 응답 결과 신호 + 세션 흐름** 기반 분석임
- IP가 수천 개로 분산돼도 공격의 본질은 **반복성, 시간 밀도, 자동화 패턴, 정상 대비 편차**에 있음

즉, 인증키 유출 이후 Brute Force 탐지에서 PLURA-XDR의 차별성은  
“유효한 키를 가진 공격자”까지 포함해 설명 가능한 2차 방어선이라는 점입니다.

이 부분은 단순한 WAF 차단이나 IP 블랙리스트 설명보다,  
실제 기업 운영 환경에서 더 현실적인 운영 시나리오에 가깝습니다.  
이러한 구조는 쿠팡과 같은 대규모 서비스 환경에서도 동일하게 유효합니다.

## 제로데이 관점

제로데이 대응에서도 두 제품의 해석 위치는 다릅니다.

SentinelOne은 엔드포인트 측면에서  
비정상 프로세스, 메모리 행위, 파일 행위, 권한 변화 등을 기반으로 후속 행위를 보는 구조입니다.

PLURA-XDR은 여기에 더해  
**웹 요청 본문과 웹 응답 본문/결과 신호를 먼저 분석**하고,  
그 후 서버 행위와 연결합니다.

따라서 제로데이 또는 비정형 웹 공격에 대해 다음과 같은 설명이 가능합니다.

- 어떤 비정상 입력이 들어왔는가
- 응답 결과가 어떤 변화를 보였는가
- 이후 서버에서 어떤 행위가 이어졌는가
- 그 행위에 대해 어떤 자동 대응이 작동했는가
- 포렌식 증적이 어떻게 함께 확보되었는가

즉, 제로데이를 단순히 후속 악성 행위로만 보는 것이 아니라  
**웹 구간의 이상 징후와 서버 구간의 후속 행위, 자동 대응, 포렌식을 함께 해석**할 수 있다는 점에서 차이가 있습니다.

이 관점은 일반 기업뿐 아니라, 쿠팡과 같은 대규모 웹/API 서비스 환경에서도 더욱 의미가 커집니다.

## 웹셸 업로드 이후 LOLBAS / GTFOBins 전환

실제 공격 흐름에서는 웹 취약점 악용 또는 웹셸 업로드가 끝점이 아니라 시작점인 경우가 많습니다.  
이후 공격자는 운영체제 기본 도구를 이용해 탐지를 우회하고 내부 행위를 지속할 수 있습니다.

대표적으로는 다음과 같은 전환이 가능합니다.

### Windows 계열
- `powershell.exe`
- `mshta.exe`
- `rundll32.exe`
- `regsvr32.exe`
- `wmic.exe`
- `certutil.exe`

### Linux 계열
- `bash`
- `sh`
- `curl`
- `wget`
- `python`
- `nc`
- 기타 GTFOBins 계열 도구

이 구간의 차이는 다음처럼 정리할 수 있습니다.

#### SentinelOne 관점
웹셸 업로드 이후 서버에서 발생한 후속 프로세스나 행위를 엔드포인트 관점에서 탐지하는 구조입니다.

#### PLURA-XDR 관점
PLURA-XDR은 다음 흐름을 함께 설명할 수 있습니다.

1. 웹 요청 또는 업로드 구간에서 비정상 입력, 파일 업로드, 응답 변화를 식별
2. 해당 이벤트를 인증/세션 흐름과 연결
3. 이후 서버 감사 로그 및 행위 로그에서 LOLBAS / GTFOBins 실행 정황 확인
4. 상관분석을 통해 웹 업로드 → 웹셸 실행 → 기본 도구 악용 → 후속 행위 확장으로 이어지는 흐름 설명
5. 자동 대응 및 포렌식 확보까지 연결

즉, 차이는 단순히 서버에서 도구가 실행되었는가가 아니라,  
**그 도구 실행이 어떤 웹 이벤트에서 시작되었는가를 연결해 설명할 수 있는가**에 있습니다.

이러한 설명 구조는 일반 기업에도 유효하며, 쿠팡과 같은 대규모 서비스 환경에서는 그 필요성이 더욱 분명해집니다.

## 웹부터 포렌식까지 자동 연결되는 구조

PLURA-XDR의 핵심 구조는  
단일 기능이 아니라 **웹부터 포렌식까지 자동 연결되는 운영 체계**입니다.

이를 단순화하면 다음과 같습니다.

### 1. 웹 구간
- 요청 본문 분석
- 응답 본문 또는 결과 신호 분석
- 제로데이 및 크리덴셜 스터핑 징후 식별
- 웹셸 업로드 또는 비정상 입력 정황 식별

### 2. XDR 상관분석
- 웹 이벤트와 인증/세션 이상행위 연결
- 기존 보안 이벤트와 상관분석
- MITRE ATT&CK 기준 행위 해석

### 3. 서버 구간
- 프로세스, 파일, 네트워크, 계정 행위 분석
- 감사 정책 기반 로그 분석
- 웹셸, LOLBAS, GTFOBins, 확산, 랜섬웨어/APT 후속 행위 식별

### 4. 자동 대응
- 제한
- 추가 인증
- 차단/격리
- 알림 연계

### 5. 포렌식
- 탐지와 동시에 증적 확보
- 사고 이후 별도 수집이 아니라 운영 흐름 안에서 증적 정리

즉, PLURA-XDR은  
**웹 → 인증/세션 → 서버 → 자동 대응 → 포렌식**을 하나의 운영 체계로 연결하는 구조입니다.

이 부분은 “모든 것을 다 잘한다”는 식의 주장보다,  
일반 기업 환경에서도 **조사 경로를 줄이고 사건 단위 설명을 가능하게 하는 운영 구조**로 설명하는 것이 더 적절합니다.  
그리고 쿠팡과 같은 대규모 서비스 환경에서는 이러한 운영 구조의 가치가 더욱 커집니다.

## 서버 감사 정책 기반 로그 분석

웹 공격은 종종 서버 행위로 이어집니다.

- 프로세스 실행
- 파일 생성 또는 변경
- 권한 상승 시도
- LOLBAS 악용
- GTFOBins 악용
- 내부 확산 전조

PLURA-XDR은 서버 행위 분석에서  
일반 이벤트 수집 수준이 아니라  
**감사 정책 기반 로그 분석**을 통해 운영체제 수준의 행위를 해석하는 구조를 함께 설명할 수 있습니다.

그리고 이 영역 역시 당사 특허 기술 범위로 제시할 수 있습니다.  
다만 이 부분은 과도한 강조보다,  
제품 비교 관점에서의 기술 신뢰 요소로 보는 것이 적절합니다.

## PoC에서 확인할 항목

이 문서는 구조적 차이를 설명하는 자료이며, 실제 도입 판단은 PoC로 검증하는 것이 적절합니다.

일반적인 기업 환경에서는 다음 항목이 우선 검증 대상이 될 수 있습니다.

1. **크리덴셜 스터핑 시나리오**
   - 요청 본문 + 응답 결과 + 세션 흐름 기반 탐지 설명력
   - 정상 사용자 실패와 자동화 공격 구분 가능성

2. **인증키/JWT 유출 시나리오**
   - 유효한 인증 수단을 가진 공격자에 대한 탐지 정확도
   - 분산 IP 환경에서 행위 기반 설명 가능성

3. **제로데이 웹 공격 시나리오**
   - 비정형 입력 탐지
   - 웹 이벤트와 서버 후속 행위 연결

4. **웹셸 → LOLBAS / GTFOBins 전환 시나리오**
   - 웹 업로드부터 서버 도구 악용까지 사건 단위 설명 가능성
   - 자동 대응 및 포렌식 확보 연계

5. **운영성**
   - Incident timeline 설명력
   - 자동 대응 속도
   - 오탐/미탐 수준
   - 운영 콘솔 활용성
   - 서비스 규모 및 트래픽 증가 상황에서의 성능과 확장성

특히 쿠팡과 같은 대규모 웹/API 서비스 환경에서는 위 항목의 중요도가 더욱 커질 수 있습니다.

## 최종 정리

이 비교에서 핵심은  
전체 XDR 시장에서의 절대적 우열을 선언하는 것이 아닙니다.

비교의 초점은 다음입니다.

- SentinelOne은 엔드포인트 중심 탐지와 자동화 구조를 가진다.
- PLURA-XDR은 일반적인 기업 환경에서  
  웹 요청·응답, 인증/세션 흐름, 서버 행위, 자동 대응, 포렌식을 하나의 흐름으로 연결하는 분석 축을 가진다.
- 특히 크리덴셜 스터핑, 인증키 기반 Brute Force, 제로데이, 웹셸 이후 LOLBAS / GTFOBins 전환 시나리오에서  
  PLURA-XDR은 더 직접적인 설명 구조를 제공할 수 있다.
- 이러한 강점은 일반 기업뿐 아니라, 쿠팡과 같은 대규모 웹/API 서비스 환경에서도 더욱 분명하게 드러날 수 있다.

따라서 이 문서의 결론은  
“PLURA가 모든 영역에서 더 우수하다”가 아니라,  
**실제 기업의 공격 표면과 운영 구조에서는 PLURA-XDR이 보완적이면서도 일부 핵심 구간에서는 더 직접적인 설명력을 제공할 수 있다**는 것입니다.
